0x09 · malloc(), Heap e Memory Leaks

// Alocação dinâmica, ciclo de vida do Heap, fragmentação e free()

Por que o Heap existe?

A Stack tem duas limitações fundamentais: tamanho fixo (típico: 8MB) e ciclo de vida vinculado ao escopo da função. Se você precisa de um array cujo tamanho só é conhecido em runtime, ou de dados que devem sobreviver ao retorno de uma função, precisa da memória dinâmica — o Heap.

O Heap é uma região do espaço de endereçamento do processo que cresce para cima (endereços crescentes) conforme solicitado. O gerenciamento é manual: você pede blocos com malloc() e deve devolvê-los com free().

malloc() — Memory ALLOCate

cmalloc_basico.c
#include <stdio.h>
#include <stdlib.h>   /* malloc, calloc, realloc, free */

int main(void) {
    /* Aloca 10 ints no heap (10 * 4 = 40 bytes) */
    int *arr = (int*) malloc(10 * sizeof(int));

    /* SEMPRE verifique o retorno — malloc pode retornar NULL */
    if (arr == NULL) {
        fprintf(stderr, "[ERRO] Falha na alocação.\n");
        return 1;
    }

    /* Inicializa (malloc NÃO zera a memória — conteúdo é lixo) */
    for (int i = 0; i < 10; i++) {
        arr[i] = i * i;
    }

    for (int i = 0; i < 10; i++) {
        printf("arr[%d] = %d\n", i, arr[i]);
    }

    free(arr);  /* devolve o bloco ao heap — OBRIGATÓRIO */
    arr = NULL; /* boa prática: evita use-after-free */
    return 0;
}

calloc e realloc

ccalloc_realloc.c
/* calloc: aloca E zera a memória (n elementos de tamanho size) */
int *zeros = (int*) calloc(10, sizeof(int));
/* zeros[0..9] == 0 garantido */

/* realloc: redimensiona um bloco existente */
int *arr = (int*) malloc(5 * sizeof(int));
if (!arr) { /* handle error */ }

/* Cresce para 10 elementos — pode mover o bloco na memória! */
int *novo = (int*) realloc(arr, 10 * sizeof(int));
if (!novo) {
    free(arr); /* realloc falhou — arr ainda é válido */
    return 1;
}
arr = novo; /* arr antigo pode ter sido movido/liberado */

⚠ Padrão perigoso com realloc: arr = realloc(arr, novo_tamanho) — se realloc falhar, retorna NULL e arr perde a referência ao bloco original → memory leak. Sempre use uma variável temporária para o retorno do realloc.

O que o heap manager faz internamente

A libc mantém metadados junto com cada bloco alocado. Antes dos bytes que você recebe de malloc há um cabeçalho (header) que guarda o tamanho do bloco e flags de status. Quando você chama free(ptr), a libc usa esses metadados para marcar o bloco como disponível e possivelmente fundir com blocos adjacentes livres (coalescing).

Heap (baixo→alto)
[ header ]sz=40|used← metadados do bloco A
[ data ]10 ints← ptr retornado por malloc ↑
[ header ]sz=24|free← bloco B (liberado)
[ data ]???← lixo (livre)
[ header ]sz=16|used← bloco C (em uso)
brk ────← topo atual do heap

Memory Leaks — vazamento de memória

Um memory leak ocorre quando um bloco alocado com malloc perde todos os ponteiros que o referenciam antes de ser liberado com free. O bloco permanece reservado até o processo terminar — inútil e inacessível.

cmemory_leak.c
void funcao_com_leak(void) {
    int *p = (int*) malloc(1024 * sizeof(int));
    if (!p) return;

    /* ... usa p ... */

    return; /* BUG: esqueceu free(p)! 4KB vazam por chamada. */
}

int main(void) {
    for (int i = 0; i < 1000000; i++) {
        funcao_com_leak(); /* ~4GB de leak! */
    }
    return 0;
}

Double-Free e Use-After-Free

cheap_bugs.c
int *p = (int*) malloc(sizeof(int));
*p = 42;

free(p);
free(p);  /* DOUBLE-FREE → corrompe estruturas internas do heap */

*p = 1;   /* USE-AFTER-FREE → undefined behavior, potencial exploit */

⚠ Heap exploitation: Double-free e use-after-free são classes de vulnerabilidade exploráveis. Um atacante pode manipular as estruturas internas do heap allocator para forjar um write-what-where primitive. Ferramentas: valgrind ./binario para detectar leaks e usos inválidos; -fsanitize=address para detecção em compile-time.

// Detecção automática: valgrind --leak-check=full --show-leak-kinds=all ./programa — reporta cada bloco perdido com o stack trace de onde foi alocado. Essencial em qualquer projeto C com alocação dinâmica.
