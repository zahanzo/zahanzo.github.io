0x07 · Operador & e a RAM

// Endereços de memória, o operador unário & e leitura do mapa de memória

O que é um endereço de memória?

A RAM é organizada como um enorme array de bytes. Cada byte tem um número único — seu endereço. Em um sistema de 64 bits, os endereços são inteiros de 64 bits (mas na prática, apenas 48-57 bits são usados, dependendo da CPU e S.O.).

Quando você declara int x = 42;, o compilador reserva 4 bytes consecutivos em alguma região da memória (stack, no caso de variáveis locais) e associa o nome x ao primeiro desses bytes. O endereço de x é o número desse primeiro byte.

O operador unário & — "endereço de"

O operador & (ampersand), quando aplicado a uma variável, retorna o endereço de memória onde ela está armazenada. O resultado é um ponteiro — um inteiro que representa uma posição na RAM.

cendereco_demo.c
#include <stdio.h>

int main(void) {
    int    x  = 42;
    double pi = 3.14159;
    char   c  = 'A';

    printf("valor  de x  : %d\n",   x);
    printf("endereço de x  : %p\n", (void*) &x);

    printf("endereço de pi : %p\n", (void*) &pi);
    printf("endereço de c  : %p\n", (void*) &c);

    /* Note que pi está 8 bytes abaixo de x (double = 8 bytes)
       e c está 1 byte abaixo de pi (char = 1 byte).
       O compilador pode inserir padding — os offsets podem variar. */
    return 0;
}

Saída típica no Linux x86-64:

outputstdout
valor  de x    : 42
endereço de x  : 0x7ffd3a4c5b2c
endereço de pi : 0x7ffd3a4c5b20
endereço de c  : 0x7ffd3a4c5b1f

Lendo a chave hexadecimal

0x7ffd3a4c5b2c é um endereço virtual na região da stack (endereços altos em Linux, com o prefixo 0x7f...). Note que:

x está em ...5b2c. pi está em ...5b20 — diferença de 12 bytes (4 bytes de x + 8 bytes de pi com alinhamento). c está em ...5b1f — 1 byte abaixo de pi.

0x7fff...5b2c2A 00 00 00← x = 42 (0x2A, little-endian)
0x7fff...5b206E 86 1B F0 F9 21 09 40← pi = 3.14159 (IEEE 754)
0x7fff...5b1f41← c = 'A' (0x41 em ASCII)

Little-Endian — a ordem dos bytes na RAM

Arquiteturas x86/x86-64 usam little-endian: o byte menos significativo (LSB) é armazenado no endereço mais baixo. Então o inteiro 42 (0x0000002A) é armazenado como: [2A] [00] [00] [00] nos bytes consecutivos — do LSB para o MSB.

Em protocolos de rede (TCP/IP), usa-se big-endian (network byte order). As funções htonl() e ntohl() convertem entre os formatos. Esse detalhe causou bugs históricos em implementações de protocolos.

Inspecionando endereços com /proc/maps

shellLinux — mapa de memória de um processo
# Encontre o PID do seu processo e leia o mapa
cat /proc/$(pidof seu_programa)/maps

# Saída típica:
# 00400000-00401000 r-xp  .text  (código executável)
# 00601000-00602000 rw-p  .data  (dados globais)
# 7f8b1234-7f8c5678 r-xp  libc.so (biblioteca dinâmica)
# 7ffd0000-7fff0000 rw-p  [stack]

// ASLR — Address Space Layout Randomization: Se você rodar o mesmo programa duas vezes, os endereços da stack serão diferentes. O Linux randomiza o layout do espaço de endereçamento em cada execução para dificultar exploits que dependem de endereços fixos. Desative para estudo: echo 0 | sudo tee /proc/sys/kernel/randomize_va_space.

scanf revisitado — por que passar &variavel

Agora fica claro por que scanf("%d", &numero) precisa do &: scanf precisa saber onde escrever o valor lido. Como C passa argumentos por valor, se você passasse apenas numero, scanf receberia uma cópia do valor e não conseguiria modificar a variável original. O &numero passa o endereço — e scanf escreve diretamente naquele endereço.
