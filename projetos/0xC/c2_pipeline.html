// Pré-processador → Compilador → Assembler → Linker

A jornada do .c até o binário

Quando você executa gcc programa.c -o programa, quatro ferramentas distintas entram em ação em sequência. Entender cada estágio explica por que certos erros aparecem em momentos específicos, como os símbolos são resolvidos e o que são arquivos .o, .a e .so.

shellpipeline explícito
# Estágio 1: Pré-processador → gera .i (C puro, sem diretivas)
gcc -E programa.c -o programa.i

# Estágio 2: Compilador → gera .s (Assembly da sua arquitetura)
gcc -S programa.i -o programa.s

# Estágio 3: Assembler → gera .o (código objeto, ELF relocável)
gcc -c programa.s -o programa.o

# Estágio 4: Linker → gera binário executável final
gcc programa.o -o programa

Estágio 1 — Pré-processador (cpp)

O pré-processador é um motor de substituição de texto. Ele não entende C. Ele processa diretivas que começam com #:

#include <stdio.h> — Insere literalmente o conteúdo do arquivo header no ponto da diretiva. Nada mais. Se você rodar gcc -E em qualquer arquivo, verá o resultado: milhares de linhas de declarações que vieram dos headers do sistema.

#define MAX 100 — Substituição textual simples. MAX vira literalmente 100 no código antes da compilação. Não há tipo, não há escopo — motivo pelo qual const int é preferível em C moderno.

cpreprocessor_demo.c
#define BUFFER_SIZE 256
#define MAX(a, b) ((a) > (b) ? (a) : (b))

char buffer[BUFFER_SIZE]; /* Vira: char buffer[256]; */
int x = MAX(3, 5);       /* Vira: int x = ((3) > (5) ? (3) : (5)); */

⚠ Macro sem parênteses = bug clássico: #define DOBRO(x) x*2 — se você chamar DOBRO(3+4), vira 3+4*2 = 11, não 14. Sempre envolva macros com parênteses: #define DOBRO(x) ((x)*2).

Estágio 2 — Compilador (cc1)

O compilador é onde a inteligência acontece. Ele recebe C puro (sem diretivas) e gera Assembly específico para a arquitetura alvo (x86-64, ARM, RISC-V). É aqui que ocorre:

Análise Léxica e Sintática — tokens, árvore AST, verificação de tipos. Os erros de compilação que você vê são daqui.

Otimizações — com -O2/-O3, o compilador pode eliminar variáveis inteiras, reordenar operações, fazer inlining de funções, etc. Use -O0 para estudar o código gerado sem surpresas.

asmmain.s — Assembly x86-64 gerado pelo GCC
main:
    pushq   %rbp          ; salva base pointer do chamador
    movq    %rsp, %rbp    ; cria novo stack frame
    movl    $0, %eax      ; prepara valor de retorno: 0
    popq    %rbp          ; restaura base pointer
    ret                   ; retorna para _start → exit(0)

Estágio 3 — Assembler (as)

O assembler converte o texto Assembly em código máquina binário, gerando um arquivo .o (objeto). Este arquivo é um ELF relocável: os endereços finais ainda não foram atribuídos. Referências a funções externas (printf, por exemplo) ficam marcadas como undefined symbols — o linker resolverá depois.

Inspecione: nm programa.o — você verá U printf (undefined) e T main (text section).

Estágio 4 — Linker (ld)

O linker recebe um ou mais arquivos .o e bibliotecas (.a estáticas ou .so dinâmicas) e produz o binário final. Suas responsabilidades:

Resolução de Símbolos — encontra a implementação de cada símbolo undefined. Se printf não for encontrado em nenhum .o ou biblioteca, você recebe o erro "undefined reference to 'printf'".

Relocação — atribui endereços virtuais finais a todas as seções e corrige os offsets no código.

Layout da Memória — decide onde ficam .text, .data, .bss no espaço de endereçamento virtual do processo.

programa.c→[cpp]→programa.i
programa.i→[cc1]→programa.s
programa.s→[as] →programa.o
*.o + libs →[ld] →binário ELF executável

Linking estático vs. dinâmico

Na linkagem estática (gcc -static), o código da libc é copiado para dentro do binário. Resultado: binário autossuficiente, maior, sem dependências. Na linkagem dinâmica (padrão), o binário contém apenas referências às bibliotecas .so que serão carregadas em runtime pelo dynamic linker (ld-linux.so). Use ldd seu_binario para ver as dependências dinâmicas.
