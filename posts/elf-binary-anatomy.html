A Anatomia de um Binário:
ELF Format & Process Memory Layout

Compilar código não cria bytes mágicos — cria um contrato estruturado entre o compilador, o linker, o loader do sistema operacional e o hardware. Este post disseca o formato ELF seção por seção, rastreia cada variável C até sua região de memória correspondente e explica por que as permissões de cada segmento importam para a segurança do sistema.

Compiling code doesn't create magic bytes — it creates a structured contract between the compiler, linker, OS loader, and hardware. This post dissects the ELF format section by section, traces every C variable to its corresponding memory region, and explains why each segment's permissions are a matter of system security.

01

O Contrato Binário

Quando você digita gcc -o program main.c, o compilador não produz uma sequência arbitrária de bytes de máquina. Ele produz um arquivo no formato ELF (Executable and Linkable Format) — um contrato preciso que especifica ao kernel exatamente como o programa deve ser carregado, quais regiões precisam de quais permissões, e qual instrução deve ser executada primeiro.

When you type gcc -o program main.c, the compiler doesn't produce an arbitrary sequence of machine bytes. It produces a file in ELF (Executable and Linkable Format) — a precise contract that tells the kernel exactly how to load the program, which regions need which permissions, and which instruction must execute first.

O ELF é o formato padrão em sistemas Linux, BSD e praticamente todo sistema Unix-like moderno. Ele é usado para três tipos de objeto distintos: executáveis (ET_EXEC / ET_DYN para PIE), objetos relocáveis (ET_REL, os arquivos .o gerados pelo compilador) e shared libraries (ET_DYN).

ELF is the standard format on Linux, BSD, and virtually every modern Unix-like system. It is used for three distinct object types: executables (ET_EXEC / ET_DYN for PIE), relocatable objects (ET_REL, the .o files produced by the compiler), and shared libraries (ET_DYN).

// O cabeçalho ELF: os primeiros 64 bytes

Todo arquivo ELF começa com um cabeçalho fixo de 64 bytes (em 64-bit). Os primeiros 4 bytes são os magic bytes — 0x7F 'E' 'L' 'F' — que permitem ao kernel identificar o tipo de arquivo instantaneamente, antes de qualquer parsing. O restante do cabeçalho descreve a arquitetura alvo, os offsets das tabelas internas e o endereço virtual do ponto de entrada.

Every ELF file starts with a fixed 64-byte header (in 64-bit). The first 4 bytes are the magic bytes — 0x7F 'E' 'L' 'F' — allowing the kernel to identify the file type instantly, before any further parsing. The rest of the header describes the target architecture, offsets of internal tables, and the virtual address of the entry point.

ELF64 header layout (Elf64_Ehdr) ascii

  0x04     1    e_ident[4]     02           →  EI_CLASS:   ELFCLASS64 (64-bit)
  0x05     1    e_ident[5]     01           →  EI_DATA:    ELFDATA2LSB (little-endian)
  0x06     1    e_ident[6]     01           →  EI_VERSION: EV_CURRENT
  0x07     1    e_ident[7]     00           →  EI_OSABI:   ELFOSABI_NONE (System V)
  0x10     2    e_type         02 00        →  ET_EXEC (executable) | 03=ET_DYN (PIE/so)
  0x12     2    e_machine      3e 00        →  EM_X86_64
  0x18     8    e_entry        <addr>       →  virtual address of _start (NOT main!)
  0x20     8    e_phoff        <offset>     →  offset of Program Header Table in file
  0x28     8    e_shoff        <offset>     →  offset of Section Header Table in file
  0x30     4    e_flags        00 00 00 00  →  processor-specific flags (none for x86_64)
  0x34     2    e_ehsize       40 00        →  size of this header = 64 bytes
  0x36     2    e_phentsize    38 00        →  size of one Program Header entry = 56 bytes
  0x38     2    e_phnum        <N>          →  number of Program Header entries
  0x3a     2    e_shentsize    40 00        →  size of one Section Header entry = 64 bytes

e_entry aponta para _start, não para main()

O ponto de entrada do processo é _start, fornecido pela libc (crt1.o). Ele inicializa o runtime C — configura o frame inicial, lê argc/argv/envp da stack, chama __libc_start_main(), que por sua vez chama main(). Quando main() retorna, __libc_start_main() chama exit(). O kernel nunca "vê" main() — ele só conhece e_entry.

The process entry point is _start, provided by the libc (crt1.o). It initializes the C runtime — sets up the initial frame, reads argc/argv/envp from the stack, calls __libc_start_main(), which in turn calls main(). When main() returns, __libc_start_main() calls exit(). The kernel never "sees" main() — it only knows e_entry.

02

Seções: a Visão do Linker

As seções são a unidade de organização do linker. Cada seção agrupa um tipo específico de dados ou código e carrega metadados sobre endereço, tamanho, tipo e flags de permissão. O linker as usa para construir o executável final; o loader as usa (via segmentos) para mapear o arquivo em memória.

Sections are the linker's unit of organization. Each section groups a specific type of data or code and carries metadata about address, size, type, and permission flags. The linker uses them to construct the final executable; the loader uses them (via segments) to map the file into memory.

principais seções e o que pertence a cada uma ascii

  ─────────── ────────── ─────  ──────────────────────────────────────────────────────
  .text       PROGBITS   AX     Compiled machine code (function bodies)
  .rodata     PROGBITS   A      Read-only data: string literals, const global arrays,
                                switch jump tables, floating-point constants
  .data       PROGBITS   WA     Initialized global/static variables (non-zero at startup)
  .bss        NOBITS     WA     Uninitialized global/static variables — zero-filled at

  .got        PROGBITS   WA     Global Offset Table: resolved symbol addresses (runtime)
  .got.plt    PROGBITS   WA     GOT entries for PLT (filled in by dynamic linker)
  .dynamic    DYNAMIC    WA     Dynamic linking metadata (DT_NEEDED, DT_SYMTAB, etc.)
  .symtab     SYMTAB     —      Full symbol table (stripped in release builds)
  .dynsym     DYNSYM     A      Minimal symbol table for dynamic linking (kept always)

  .shstrtab   STRTAB     —      Section name strings (the "index" of sections)

          X = Executable
          W = Writable

// .bss — sem espaço em disco

A seção .bss (Block Started by Symbol) é o detalhe mais contra-intuitivo do formato ELF. Variáveis globais não-inicializadas são garantidas de serem zero pelo padrão C. Em vez de armazenar megabytes de zeros no arquivo binário, o linker simplesmente registra o tamanho que .bss precisa ter. O kernel, ao carregar o programa, mapeia essa região como páginas anônimas zero-filled via mmap(MAP_ANONYMOUS). O resultado é um executável menor no disco e o mesmo comportamento em runtime.

The .bss section (Block Started by Symbol) is the most counter-intuitive detail of the ELF format. Uninitialized global variables are guaranteed to be zero by the C standard. Rather than storing megabytes of zeros in the binary file, the linker simply records the size that .bss needs. When loading the program, the kernel maps that region as anonymous zero-filled pages via mmap(MAP_ANONYMOUS). The result is a smaller on-disk binary with identical runtime behavior.

// .rodata vs .data

A distinção entre .rodata e .data existe para que o kernel possa mapear literais de string e constantes em páginas somente-leitura. Tentar escrever em .rodata via um ponteiro gera um SIGSEGV imediato — não porque o compilador checou algo, mas porque a própria página de memória não tem o bit de escrita habilitado na PTE. É a MMU fazendo o enforcement.

The distinction between .rodata and .data exists so the kernel can map string literals and constants into read-only pages. Attempting to write to .rodata via a pointer produces an immediate SIGSEGV — not because the compiler checked anything, but because the memory page itself does not have the write bit set in the PTE. It is the MMU doing the enforcement.

rodata write fault — runtime, not compile-time c 
const char *msg = "hello";  /* ptr in .data, string literal in .rodata */

/* This compiles fine with a cast — the compiler cannot always detect it */
((char *)msg)[0] = 'H';
/* SIGSEGV at runtime: write to read-only page (PTE.W = 0)              */
/* Reported by kernel as: signal 11 (SIGSEGV), si_code=SEGV_ACCERR     */

Segmentos: a Visão do Loader

As seções existem para o linker. Os segmentos (definidos pelos Program Headers) existem para o loader. A diferença é fundamental: seções são granulares e detalhadas; segmentos são coarsos e orientados ao mapeamento de memória. O loader do kernel não processa seções individuais — ele itera sobre os Program Headers e chama mmap() para cada segmento do tipo PT_LOAD.

Sections exist for the linker. Segments (defined by Program Headers) exist for the loader. The difference is fundamental: sections are granular and detailed; segments are coarse and oriented toward memory mapping. The kernel loader does not process individual sections — it iterates over the Program Headers and calls mmap() for each PT_LOAD segment.

// O que o kernel faz com cada PT_LOAD

Para cada segmento PT_LOAD, o kernel executa essencialmente:

For each PT_LOAD segment, the kernel essentially executes:

kernel ELF loader — simplified mmap per PT_LOAD c 
/* For each PT_LOAD segment in the program header table: */
for (int i = 0; i < ehdr.e_phnum; i++) {
    if (phdr[i].p_type != PT_LOAD) continue;

    int prot = 0;
    if (phdr[i].p_flags & PF_R) prot |= PROT_READ;
    if (phdr[i].p_flags & PF_W) prot |= PROT_WRITE;
    if (phdr[i].p_flags & PF_X) prot |= PROT_EXEC;

    /* Map the file content (FileSiz) at the virtual address */
    mmap(phdr[i].p_vaddr,          /* virtual address (page-aligned) */
         phdr[i].p_filesz,         /* bytes to copy from file        */
         prot,
         MAP_PRIVATE | MAP_FIXED,
         elf_fd,                   /* file descriptor of the binary  */
         phdr[i].p_offset);        /* offset within the ELF file     */

    /* If MemSiz > FileSiz, the remainder is .bss — map as zero pages */
    if (phdr[i].p_memsz > phdr[i].p_filesz) {
        size_t bss_size = phdr[i].p_memsz - phdr[i].p_filesz;
        mmap(phdr[i].p_vaddr + phdr[i].p_filesz,
             bss_size,
             prot,
             MAP_PRIVATE | MAP_FIXED | MAP_ANONYMOUS,  /* zero-filled */
             -1, 0);
    }
}

PT_INTERP: o loader antes do loader

O segmento PT_INTERP contém o path do dynamic linker (/lib64/ld-linux-x86-64.so.2). Quando o kernel detecta esse segmento, ele não salta para e_entry diretamente. Ele primeiro carrega e executa o dynamic linker, que então: (1) mapeia todas as shared libraries listadas em DT_NEEDED, (2) resolve símbolos entre elas, (3) aplica relocações, (4) executa os construtores .init_array e finalmente (5) transfere controle para e_entry do executável original. Para um binário estático (compilado com -static), PT_INTERP não existe e o kernel vai direto para e_entry.

The PT_INTERP segment contains the path to the dynamic linker (/lib64/ld-linux-x86-64.so.2). When the kernel detects this segment, it does not jump to e_entry directly. It first loads and executes the dynamic linker, which then: (1) maps all shared libraries listed in DT_NEEDED, (2) resolves symbols between them, (3) applies relocations, (4) runs .init_array constructors, and finally (5) transfers control to the original executable's e_entry. For a statically linked binary (-static), PT_INTERP does not exist and the kernel jumps directly to e_entry.

C → Memória: rastreando variáveis com readelf e nm

O exemplo abaixo foi construído especificamente para colocar ao menos uma variável em cada seção relevante. Compilado com gcc -no-pie -O0 para endereços fixos e sem otimizações que eliminariam variáveis não usadas.

The example below was specifically constructed to place at least one variable in each relevant section. Compiled with gcc -no-pie -O0 for fixed addresses and without optimizations that would eliminate unused variables.

example.c — one variable per section c 
#include <stdio.h>

/* .rodata — read-only data: string literal, never writable */
const char *greeting    = "Hello, ELF!";

/* .data — initialized global: has a non-zero value at startup */
int   counter          = 42;

/* .bss — uninitialized global: zero-filled by kernel, no disk space */
int   uninit_global;

/* .bss — zero-initialized static: same section as uninit globals */
static char buffer[1024];

/* .data — initialized static: non-zero value lives on disk */
static int  magic      = 0xDEAD;

/* .text — function code: compiled to machine instructions */
void greet(void) {
    printf("%s\n", greeting);  /* ptr in .data, string in .rodata */
    counter++;
}

int main(void) {
    greet();
    buffer[0] = (char) magic;  /* .bss write — page is RW, not RX */
    return counter;
}

05

Permissões, W^X e a Cadeia de Segurança

As permissões de cada segmento não são apenas organização — elas são a primeira linha de defesa contra classes inteiras de exploits. O princípio W^X (Write XOR Execute) estabelece que uma página de memória nunca deve ser simultaneamente gravável e executável. O ELF, o kernel e o hardware cooperam para garantir isso.

Each segment's permissions are not merely organizational — they are the first line of defense against entire classes of exploits. The W^X (Write XOR Execute) principle establishes that a memory page should never be simultaneously writable and executable. The ELF, kernel, and hardware cooperate to enforce this.

permissões por segmento e o ataque que cada uma previne ascii 
  Region     Permissions   PTE bits          Attack prevented
  ─────────  ────────────  ────────────────  ──────────────────────────────────────────
  .text      r-x           R=1  W=0  X=1     Code injection: attacker cannot modify
             (RX not W)    NX=0              code pages even after code execution
                                             Write attempt → #PF (SIGSEGV)

  .rodata    r--           R=1  W=0  X=0     Data tampering: string literals, vtables,
             (R only)      NX=1              and jump tables cannot be overwritten
                                             Execute attempt → #PF (NX violation)

  .data      rw-           R=1  W=1  X=0     Code execution via data: attacker may
  .bss       (RW not X)    NX=1              write shellcode but cannot execute it
                                             Execute attempt → #PF (NX violation)

  .got       r-- (RELRO)   R=1  W=0  X=0     GOT overwrite: after startup, GOT entries
             after startup NX=1              are made read-only. Overwriting a GOT
                                             entry to redirect function calls fails.

// A cadeia completa de proteções modernas

hardening de binário — camadas empilhadas ascii 
  NX / XD bit  (ELF p_flags PF_X → PTE bit 63 = Execute Disable)
  ├── Hardware enforcement: data pages cannot be executed even if jumped to.
  ├── Defeats: classic shellcode injection (stack/heap shellcode).
  └── Bypass: ROP (Return-Oriented Programming) — reuses existing .text gadgets.

  ASLR  (kernel: randomize_va_space, PIE binary: ET_DYN + -fPIE)
  ├── Randomizes base address of code, stack, heap, mmap at every execve().
  ├── Attacker cannot hardcode addresses of gadgets, PLT stubs, or libc symbols.
  ├── Requires PIE (Position Independent Executable): code uses RIP-relative addressing.
  └── Bypass: information leak — a single leaked pointer kills ASLR.

  RELRO  (GNU_RELRO segment + BIND_NOW / -z now)
  ├── After dynamic linker resolves all symbols, remaps .got and .init_array as R--.
  ├── Partial RELRO: .got.plt still writable (lazy binding still possible).
  ├── Full RELRO (-z now): ALL GOT entries resolved at startup, entire GOT is R--.
  └── Defeats: GOT overwrite attacks that redirect function pointers.

  Stack Canary  (gcc -fstack-protector-strong)
  ├── Compiler inserts a random value between locals and the saved return address.
  ├── Checked before RET — mismatch aborts the process.
  └── Defeat: info leak to read the canary value, then overwrite with correct value.

  CFI / CET  (Intel Control-flow Enforcement Technology, -fcf-protection)
  ├── ENDBR64 instruction marks valid indirect call/jump targets.
  ├── Shadow Stack: hardware-maintained copy of return addresses.
  └── Defeats: ROP chains and JOP (Jump-Oriented Programming).

Por que PIE torna tudo diferente

Um binário sem PIE (ET_EXEC) tem endereços absolutos hardcoded pelo linker — .text começa em 0x400000, sempre. Um binário PIE (ET_DYN, compilado com -fPIE) usa endereçamento relativo ao RIP para tudo. O kernel pode carregá-lo em qualquer base virtual aleatória a cada execução. Sem ASLR, a segurança fornecida pelo NX e RELRO é significativamente enfraquecida porque o atacante conhece todos os endereços de antemão.

A non-PIE binary (ET_EXEC) has absolute addresses hardcoded by the linker — .text always starts at 0x400000. A PIE binary (ET_DYN, compiled with -fPIE) uses RIP-relative addressing for everything. The kernel can load it at any random virtual base on each execution. Without ASLR, the security provided by NX and RELRO is significantly weakened because the attacker knows all addresses in advance.
