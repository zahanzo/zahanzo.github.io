O Paradoxo do Boot

Pressione o botão de liga do seu servidor com 64 núcleos, 512 GB de RAM e CPUs rodando a 5 GHz. No instante em que o sinal RESET# é desassertado pelo chipset, cada núcleo acorda num estado que data de 1978: 16-bit Real Mode, com um espaço de endereçamento de exatos 1 MB, sem proteção de memória, sem paginação, sem anéis de privilégio. O processador mais avançado da Intel, ao ligar, é indistinguível de um 8086 original.

Press the power button on your 64-core server with 512 GB of RAM and CPUs running at 5 GHz. The instant the RESET# signal is deasserted by the chipset, every core wakes up in a state that dates back to 1978: 16-bit Real Mode, with an address space of exactly 1 MB, no memory protection, no paging, no privilege rings. The most advanced Intel processor, on power-on, is indistinguishable from an original 8086.

Isso não é um bug. É compatibilidade retroativa forçada — a Intel não pode quebrar o boot de nenhum sistema que existiu nos últimos 46 anos. Então todo sistema operacional moderno, todo hypervisor, todo kernel precisa resolver o mesmo problema que o GRUB, o Linux e o Windows Boot Manager resolvem: acordar nesse ambiente antediluviano e escalar para o modo de 64 bits o mais rápido possível.

This is not a bug. It is forced backward compatibility — Intel cannot break the boot of any system that has existed in the last 46 years. So every modern OS, every hypervisor, every kernel must solve the same problem that GRUB, Linux, and the Windows Boot Manager solve: wake up in this antediluvian environment and scale up to 64-bit mode as fast as possible.

Real Mode (16-bit): o confinamento do 8086

O 8086 tinha um barramento de endereços de 20 bits — 2²⁰ = 1.048.576 bytes = exatos 1 MB. Os registradores eram todos de 16 bits, então o processador usava um esquema de segmentação para alcançar o espaço completo: endereço físico = segment × 16 + offset.

The 8086 had a 20-bit address bus — 2²⁰ = 1,048,576 bytes = exactly 1 MB. Registers were all 16 bits, so the processor used a segmentation scheme to reach the full space: physical address = segment × 16 + offset.

Real Mode address translation ascii 
  Segment register (16-bit):  0x1234
  Offset (16-bit):            0x5678
                              ──────────────────────────────────────
  Physical address = (0x1234 << 4) + 0x5678
                   =  0x12340       + 0x5678
                   =  0x179B8          ← 20-bit physical address

  Same offset, different segment → completely different physical address.
  No hardware enforces who can access what: ALL code accesses ALL memory.

  Maximum addressable:  0xFFFF:0xFFFF = 0xFFFF0 + 0xFFFF = 0x10FFEF
  Address bus limit:    0xFFFFF (20 bits)
  Addresses above 0xFFFFF "wrap around" to 0x00000 — the infamous A20 issue.

                  │  IVT: 256 × 4-byte vectors  │ 0x0000–0x03FF (1 KB)

// O problema da linha A20

Quando a IBM lançou o PC/AT com o 286, o barramento de endereços expandiu para 24 bits. Mas alguns programas DOS dependiam do wrap-around acima de 1 MB (onde 0xFFFF:0xFFFF + 1 voltava para 0x00000). Para manter compatibilidade, a IBM adicionou um gate elétrico na linha de endereço 20 (A20) — controlado, bizarramente, pelo controlador do teclado. Todo bootloader moderno precisa habilitar a linha A20 antes de acessar qualquer memória acima de 1 MB, caso contrário o sistema operará com a linha A20 mascarada para zero.

When IBM shipped the PC/AT with the 286, the address bus expanded to 24 bits. But some DOS programs relied on the wrap-around above 1 MB (where 0xFFFF:0xFFFF + 1 looped back to 0x00000). For compatibility, IBM added an electrical gate on address line 20 (A20) — controlled, bizarrely, by the keyboard controller. Every modern bootloader must enable the A20 line before accessing any memory above 1 MB, otherwise the system will operate with A20 masked to zero.

enable_a20 — fast A20 via port 0x92 nasm 
enable_a20:
    in   al, 0x92          ; read System Control Port A
    or   al, 0x02          ; set bit 1 = A20 enable
    and  al, 0xFE          ; clear bit 0 = do NOT reset system
    out  0x92, al          ; write back — A20 now enabled
    ret

    ; Alternative method: BIOS INT 15h, AX=2401h (call before leaving Real Mode)
    ; Alternative method: keyboard controller port 0x64/0x60 (slower, but universal)

Real Mode limitations — sem defesa nenhuma

Em Real Mode: qualquer instrução pode escrever em qualquer endereço físico — incluindo a IVT em 0x0000, a BIOS ROM, ou o código do próprio bootloader. Não há anéis de privilégio (ring 0/3 não existem). Não há paginação nem proteção de páginas. Uma string de memset(0, 0, 1024*1024) apagaria o 1 MB inteiro sem nenhuma resistência do hardware. É por isso que a saída do Real Mode é a primeira coisa que qualquer bootloader faz.

In Real Mode: any instruction can write to any physical address — including the IVT at 0x0000, the BIOS ROM, or the bootloader's own code. There are no privilege rings (ring 0/3 do not exist). There is no paging or page protection. A memset(0, 0, 1024*1024) would wipe the entire 1 MB without any hardware resistance. This is why leaving Real Mode is the first thing any bootloader does.

03

Protected Mode (32-bit): a GDT e o CR0

Para ativar o Protected Mode, o bootloader precisa cumprir um protocolo preciso em 4 passos. Pular ou inverter qualquer passo resulta em uma exceção de proteção geral (#GP) ou, mais provavelmente, em um triple fault silencioso e reset da máquina.

To activate Protected Mode, the bootloader must follow a precise 4-step protocol. Skipping or reordering any step results in a general protection fault (#GP) or, more likely, a silent triple fault and machine reset.

mandatory sequence to enter Protected Mode ascii 
  1. CLI              — disable maskable interrupts (IVT is no longer valid)
  2. LGDT [desc]      — load the GDT register with base + limit of the GDT
  3. MOV CR0, ...     — set CR0.PE (bit 0) to enable protection
  4. JMP FAR 0x08:pm  — flush the instruction pipeline with a far jump
                         loads CS with a valid code segment selector from GDT

// GDT — Global Descriptor Table

A GDT é uma tabela residente em memória onde cada entrada de 8 bytes (um segment descriptor) descreve um segmento de memória: seu endereço base, seu limite e seus atributos de acesso. O registrador GDTR é carregado com a instrução LGDT e aponta para essa tabela. O kernel não usa segmentação para isolamento (usa paginação), mas a GDT ainda precisa existir — o hardware exige descritores válidos para CS, SS e os demais registradores de segmento.

The GDT is a memory-resident table where each 8-byte entry (a segment descriptor) describes a memory segment: its base address, its limit, and its access attributes. The GDTR register is loaded with the LGDT instruction and points to this table. The kernel does not use segmentation for isolation (it uses paging), but the GDT must still exist — hardware requires valid descriptors for CS, SS, and the other segment registers.

8-byte segment descriptor layout ascii 
  63      56 55   52 51  48 47      40 39       16 15         0
  ┌─────────┬───────┬──────┬──────────┬───────────┬────────────┐
  │ Base    │ Flags │ Lim  │  Access  │   Base    │   Limit    │
  │ [31:24] │ [3:0] │[19:16│   Byte   │  [23:0]   │  [15:0]    │
  └─────────┴───────┴──────┴──────────┴───────────┴────────────┘
               │               │
               │               └─ Bit 7 (P):   Present (1 = valid)
               │                  Bits 6-5 (DPL): Privilege (0=ring0, 3=ring3)
               │                  Bit 4 (S):   1=code/data, 0=system (TSS,gate)
               │                  Bit 3 (E):   1=executable (code), 0=data
               │                  Bit 2 (DC):  Direction (data) / Conforming (code)
               │                  Bit 1 (RW):  Writable (data) / Readable (code)
               │                  Bit 0 (A):   Accessed (CPU sets on use)
               │
               └─ Bit 3 (G):   Granularity: 1=limit in 4KB units, 0=bytes
                  Bit 2 (D/B): 1=32-bit operands, 0=16-bit
                  Bit 1 (L):   1=64-bit code (Long Mode), D/B must be 0
                  Bit 0 (AVL): Available for OS use

Para o modelo flat de 32 bits (base=0, limit=4GB — o que todo sistema moderno usa), os descritores de código e dados são quase idênticos, diferindo apenas no bit E (Executable) do Access Byte. O selector que vai em CS deve apontar para um descriptor com E=1; os seletores em DS/ES/SS devem apontar para descritores com E=0.

For the 32-bit flat model (base=0, limit=4GB — what every modern system uses), the code and data descriptors are nearly identical, differing only in the E (Executable) bit of the Access Byte. The selector loaded into CS must point to a descriptor with E=1; selectors in DS/ES/SS must point to descriptors with E=0.

04

Long Mode (64-bit): paginação, PAE e EFER

A transição para Long Mode é mais complexa que a entrada em Protected Mode porque Long Mode requer paginação ativa. Você não pode habilitar o Long Mode e configurar as page tables depois — o hardware exige que as tabelas estejam prontas antes do bit PG ser setado. A sequência tem 5 passos, e a ordem é mandatória.

The transition to Long Mode is more complex than entering Protected Mode because Long Mode requires active paging. You cannot enable Long Mode and set up page tables afterward — the hardware requires the tables to be ready before the PG bit is set. The sequence has 5 steps, and the order is mandatory.

mandatory Long Mode sequence (Intel SDM §9.8.5) ascii 
  From 32-bit Protected Mode:

  1. Build paging structures in physical memory (PML4 → PDPT → PD → PT or 2MB pages)
     Load CR3 with physical address of PML4 table

  2. Set CR4.PAE (bit 5) — Physical Address Extension
     Required: Long Mode uses 4-level page tables, PAE must be on

  3. Set EFER.LME (bit 8) via WRMSR to MSR 0xC0000080
     "Long Mode Enable" — signals intent, but does NOT activate Long Mode yet

  4. Set CR0.PG (bit 31) AND ensure CR0.PE (bit 0) is still set
     THIS activates Long Mode: hardware sets EFER.LMA (bit 10) automatically

  5. JMP FAR to 64-bit code segment (GDT descriptor with L=1, D/B=0)
     Flushes the pipeline; CPU is now in IA-32e (64-bit) mode

// EFER — Extended Feature Enable Register

O EFER (MSR 0xC0000080) é o registrador que habilita features avançadas da arquitetura. Ele é acessado exclusivamente via RDMSR/WRMSR — não existe instrução direta como para CR0. Os três bits relevantes ao boot são:

The EFER (MSR 0xC0000080) is the register that enables advanced architectural features. It is accessed exclusively via RDMSR/WRMSR — there is no direct instruction as with CR0. The three bits relevant to boot are:

EFER MSR (0xC0000080) — key bits ascii 
  Bit 11 (NXE): No-Execute Enable  — enables XD/NX bit in PTEs (PTE bit 63)
  Bit 10 (LMA): Long Mode Active   — READ-ONLY: CPU sets this when PG is set with LME=1
  Bit  8 (LME): Long Mode Enable   — YOU set this in step 3 (desire, not yet active)
  Bit  0 (SCE): SYSCALL Enable     — enables SYSCALL/SYSRET instructions

  LME=1 alone does nothing.
  LME=1 + CR0.PG=1  →  CPU sets LMA=1  →  Long Mode is ACTIVE.

// Page tables mínimas para o boot

Para entrar em Long Mode, as page tables precisam fazer pelo menos um mapeamento válido para o código que está executando (o bootloader em si). A solução mais simples é um identity mapping da RAM baixa usando páginas de 2 MB (huge pages com bit PS=1 no PD entry), evitando a necessidade de uma quarta camada de tabela (PT).

To enter Long Mode, the page tables must provide at least one valid mapping for the executing code (the bootloader itself). The simplest solution is an identity mapping of low RAM using 2 MB pages (huge pages with PS=1 in the PD entry), avoiding the need for a fourth table layer (PT).

minimal paging structure for Long Mode ascii 
  Physical memory layout of page tables (placed at known addresses):

  0x1000: PML4 Table   (4 KB, 512 × 8-byte entries)
            entry[0] = 0x2003  ← points to PDPT at 0x2000, flags: P=1 RW=1
  0x2000: PDPT Table   (4 KB, 512 × 8-byte entries)
            entry[0] = 0x3003  ← points to PD at 0x3000,   flags: P=1 RW=1
  0x3000: PD Table     (4 KB, 512 × 8-byte entries)
            entry[0] = 0x0083  ← 2MB page at phys 0x000000, flags: P=1 RW=1 PS=1
            entry[1] = 0x20083 ← 2MB page at phys 0x200000, flags: P=1 RW=1 PS=1
            ...

  PDE entry flags for 2MB page:
  Bit  7 (PS):  Page Size = 1  →  this is a 2MB page, no PT needed
  Bit  1 (RW):  Read/Write = 1
  Bit  0 (P):   Present = 1
  Bits 20:21 = physical address bits [31:21] of the 2MB-aligned frame

  CR3 = 0x1000  (physical address of PML4)
  This gives an identity map: vaddr 0x000000–0x1FFFFF → paddr 0x000000–0x1FFFFF

Por que paginação é obrigatória para Long Mode?

Em Long Mode, a segmentação é efetivamente desabilitada: base e limite dos descritores de segmento são ignorados (exceto para FS/GS usados em TLS). Toda proteção de memória, isolamento entre processos e entre kernel/usuário passa exclusivamente pelas page tables. Por design, o hardware não permite Long Mode sem paginação ativa — seria um modo sem nenhuma proteção de memória operacional.

In Long Mode, segmentation is effectively disabled: the base and limit of segment descriptors are ignored (except for FS/GS used for TLS). All memory protection, process isolation, and kernel/user separation passes exclusively through the page tables. By design, hardware does not allow Long Mode without active paging — it would be a mode with no operational memory protection.

Assembly: os stubs exatos de transição

Os três blocos abaixo compõem um bootloader mínimo funcional. O código está em NASM, comentado linha por linha. A ordem é: (1) GDT para as três arquiteturas, (2) transição Real → Protected Mode, (3) transição Protected → Long Mode.

The three blocks below compose a functional minimal bootloader. The code is in NASM, commented line by line. The order is: (1) GDT for all three architectures, (2) Real → Protected Mode transition, (3) Protected → Long Mode transition.

gdt.asm — GDT definitions for all three modes nasm 
align 8
gdt_start:

; ── Descriptor 0x00: Null (mandatory — CPU requires entry 0 to be null) ──
    dq 0x0000000000000000

; ── Descriptor 0x08: 32-bit Code Segment ─────────────────────────────────
; Base=0x000000, Limit=0xFFFFF (×4KB = 4GB), DPL=0, Execute+Read
    dw 0xFFFF     ; Limit[15:0]  = 0xFFFF
    dw 0x0000     ; Base [15:0]  = 0x0000
    db 0x00       ; Base [23:16] = 0x00
    db 0x9A       ; Access: P=1 DPL=0 S=1 E=1 DC=0 RW=1 A=0 → 1001 1010
    db 0xCF       ; Flags+Lim[19:16]: G=1 D/B=1 L=0 AVL=0 | 1111 → 1100 1111
    db 0x00       ; Base [31:24] = 0x00

; ── Descriptor 0x10: 32-bit Data Segment ─────────────────────────────────
; Same as code, but E=0 (data), RW=1 (writable)
    dw 0xFFFF
    dw 0x0000
    db 0x00
    db 0x92       ; Access: P=1 DPL=0 S=1 E=0 DC=0 RW=1 A=0 → 1001 0010
    db 0xCF
    db 0x00

; ── Descriptor 0x18: 64-bit Code Segment (Long Mode) ─────────────────────
; Base and Limit are IGNORED in 64-bit mode (flat model enforced by hardware)
; Critical: L=1 (bit 1 of flags byte) and D/B=0
    dw 0x0000     ; Limit (ignored)
    dw 0x0000     ; Base  (ignored)
    db 0x00       ; Base  (ignored)
    db 0x9A       ; Access: P=1 DPL=0 S=1 E=1 DC=0 RW=1 A=0
    db 0xA0       ; Flags: G=1 D/B=0 L=1 AVL=0 | 0000 → 1010 0000
    db 0x00

; ── Descriptor 0x20: 64-bit Data Segment ─────────────────────────────────
    dw 0x0000
    dw 0x0000
    db 0x00
    db 0x92       ; Access: P=1 DPL=0 S=1 E=0 DC=0 RW=1 A=0
    db 0x00       ; Flags: G=0 D/B=0 L=0 (data segments irrelevant in LM)
    db 0x00

gdt_end:

; GDTR descriptor — loaded by LGDT instruction
gdt_descriptor:
    dw gdt_end - gdt_start - 1   ; 16-bit limit (size - 1)
    dq gdt_start                  ; 64-bit base address
real_to_pm.asm — Real Mode → Protected Mode transition nasm 
bits 16
org  0x7C00             ; BIOS loads the MBR here

start:
    ; ── Step 0: Disable interrupts ────────────────────────────────────────
    ; The IVT at 0x0000 is only valid in Real Mode. Once we switch,
    ; any interrupt before the IDT is loaded would be catastrophic.
    cli

    ; ── Step 0b: Enable A20 line ──────────────────────────────────────────
    in   al, 0x92
    or   al, 0x02
    and  al, 0xFE
    out  0x92, al

    ; ── Step 1: Load the GDT ──────────────────────────────────────────────
    ; LGDT reads a 6-byte structure: 2-byte limit + 4-byte (or 8-byte) base
    lgdt [gdt_descriptor]

    ; ── Step 2: Set PE bit in CR0 ─────────────────────────────────────────
    ; CR0 cannot be loaded with an immediate — must go through a GPR
    mov  eax, cr0
    or   eax, 0x1          ; set bit 0 = PE (Protection Enable)
    mov  cr0, eax
    ; We are NOW in Protected Mode — but CS still has the Real Mode value.
    ; The processor is in a hybrid state until the far jump.

    ; ── Step 3: Far jump — flush pipeline + load CS from GDT ─────────────
    ; 0x08 = selector for GDT entry 1 (32-bit code segment)
    ; The far jump atomically:  loads CS ← 0x08  AND  EIP ← pm32_entry
    jmp  0x08:pm32_entry

bits 32
pm32_entry:
    ; ── Step 4: Reload all data segment registers ─────────────────────────
    ; 0x10 = selector for GDT entry 2 (32-bit data segment)
    mov  ax,  0x10
    mov  ds,  ax
    mov  es,  ax
    mov  fs,  ax
    mov  gs,  ax
    mov  ss,  ax
    mov  esp, 0x90000       ; set up a stack below the Real Mode map

    ; 32-bit Protected Mode is active.
    ; Full 4GB address space, privilege rings, hardware exceptions available.
    call enter_long_mode   ; proceed to 64-bit
pm_to_lm.asm — Protected Mode → Long Mode transition nasm 
bits 32

enter_long_mode:
    ; ── Step 1: Build identity-mapped page tables ─────────────────────────
    ; PML4 at 0x1000, PDPT at 0x2000, PD at 0x3000 (all physically allocated)

    ; Zero all three tables (3 pages × 4096 bytes)
    mov  edi, 0x1000
    xor  eax, eax
    mov  ecx, (3 * 4096) / 4
    rep  stosd

    ; PML4[0] → PDPT: flags P=1(present) RW=1(writable)
    mov  dword [0x1000], 0x2003

    ; PDPT[0] → PD: flags P=1 RW=1
    mov  dword [0x2000], 0x3003

    ; PD[0] → 2MB huge page at phys 0x000000: P=1 RW=1 PS=1(2MB page)
    mov  dword [0x3000], 0x000083
    ; PD[1] → 2MB huge page at phys 0x200000: maps the next 2MB
    mov  dword [0x3008], 0x200083

    ; Load CR3 with physical address of PML4
    mov  eax, 0x1000
    mov  cr3, eax           ; CR3 = PML4 base — also flushes TLB

    ; ── Step 2: Enable PAE (Physical Address Extension) ───────────────────
    ; PAE extends physical addresses to 52 bits — required for Long Mode
    mov  eax, cr4
    or   eax, (1 << 5)       ; CR4.PAE = bit 5
    mov  cr4, eax

    ; ── Step 3: Set LME in EFER MSR ───────────────────────────────────────
    ; RDMSR/WRMSR: ECX = MSR address, EDX:EAX = value
    mov  ecx, 0xC0000080    ; EFER MSR address
    rdmsr                   ; read current EFER into EDX:EAX
    or   eax, (1 << 8)       ; bit 8 = LME (Long Mode Enable)
    or   eax, (1 << 0)       ; bit 0 = SCE (SYSCALL/SYSRET enable)
    or   eax, (1 << 11)      ; bit 11 = NXE (No-Execute Enable)
    wrmsr                   ; write back — LME set, but NOT yet active

    ; ── Step 4: Enable Paging + keep PE set in CR0 ────────────────────────
    ; Setting PG while LME=1 atomically activates Long Mode (sets EFER.LMA)
    mov  eax, cr0
    or   eax, (1 << 31) | (1 << 0)  ; PG=1 | PE=1
    mov  cr0, eax
    ; EFER.LMA is now 1. Long Mode is active.
    ; But CS still has the 32-bit descriptor — we are in "compatibility mode".

    ; ── Step 5: Far jump to 64-bit code segment ───────────────────────────
    ; 0x18 = GDT entry 3 (64-bit code descriptor with L=1, D/B=0)
    ; After this jump, CS is loaded with the 64-bit descriptor.
    jmp  0x18:lm64_entry

O estado entre CR0.PG=1 e o far jump

Entre o momento em que CR0.PG é setado e a execução do JMP FAR 0x18:, o CPU está em um estado híbrido chamado Compatibility Mode (um submodo do Long Mode onde código de 32 bits ainda pode executar). O registrador CS ainda carrega o seletor de 32 bits. Cada instrução executada nesse gap é decodificada como código de 32 bits usando o descritor de 32 bits no GDT. O far jump é o momento em que CS é recarregado com o descritor de 64 bits (L=1) e o processador começa a decodificar instruções como código de 64 bits. Qualquer instrução de 64 bits executada antes desse jump provavelmente causará um triple fault.

Between the moment CR0.PG is set and the execution of JMP FAR 0x18:, the CPU is in a hybrid state called Compatibility Mode (a sub-mode of Long Mode where 32-bit code can still run). The CS register still holds the 32-bit selector. Every instruction executed in this gap is decoded as 32-bit code using the 32-bit descriptor in the GDT. The far jump is the moment CS is reloaded with the 64-bit descriptor (L=1) and the processor begins decoding instructions as 64-bit code. Any 64-bit instruction executed before this jump will likely cause a triple fault.
