01

O Problema Impossível

Aqui está o desafio. Um array de inteiros onde todo número aparece exatamente 3 vezes, exceto um único intruso que aparece apenas 1 vez. Encontre o intruso. Restrições absolutas: O(n) tempo e O(1) espaço.

Here is the challenge. An array of integers where every number appears exactly 3 times, except one lone intruder that appears exactly once. Find the intruder. Absolute constraints: O(n) time and O(1) space.

the problem cpp 
/* Every element appears exactly 3 times, except ONE anomaly */
int arr[] = { 2, 2, 3, 2 };          // answer: 3
int arr[] = { 5, 5, 7, 5, 9, 9, 9 };  // answer: 7
int arr[] = { 0, 1, 0, 1, 0, 1, 99 }; // answer: 99

// Rules:                                                        */
// ✗ No HashMap, unordered_map, std::map — those are O(n) space  */
// ✗ No sorting — that is O(n log n) time                        */
// ✗ No extra array or buffer — still O(n) space                 */
// ✓ Two int variables. That is it. Period.                       */

// Por que as soluções óbvias falham?

A solução "óbvia" é um HashMap contando frequências. Funciona, mas viola O(1) espaço: o mapa cresce com n. Sorting funciona em O(1) espaço extra mas viola O(n) tempo. E o truque XOR elegante que funciona para "número duplicado" (a ^ a == 0)? Não funciona aqui — XOR é módulo-2, e nosso problema é módulo-3. a ^ a ^ a == a, não zero.

The "obvious" solution is a HashMap counting frequencies. It works, but violates O(1) space: the map grows with n. Sorting works in O(1) extra space but violates O(n) time. And the elegant XOR trick that works for "duplicate number" (a ^ a == 0)? It does not work here — XOR is modulo-2, and our problem is modulo-3. a ^ a ^ a == a, not zero.

why plain XOR doesn't work ascii 
  Mod-2 problem (each number appears TWICE):
    XOR all elements → duplicates cancel (a ^ a = 0) → unique survives ✓

  Mod-3 problem (each number appears THREE times):
    XOR all elements → a ^ a ^ a = a (does NOT cancel!) ✗

  We need a counter that resets after 3, not after 2.
  That requires a 2-bit state machine — and that is exactly what we build.

// A quebra de paradigma que está chegando

A solução que vamos construir não compara números entre si. Ela não pergunta "esse número já apareceu antes?". Ela trata cada número como um sinal físico que altera o estado de duas variáveis ao mesmo tempo, independentemente de qualquer outro número no array.

The solution we are about to build does not compare numbers with each other. It does not ask "has this number appeared before?". It treats each number as a physical signal that simultaneously changes the state of two variables, completely independently of any other number in the array.

02

A Virada Paradigmática

Todo mundo pensa "horizontalmente" — percorremos o array da esquerda para a direita, comparando elementos, construindo mapas. Mas o segredo desta solução é pensar verticalmente.

Everyone thinks "horizontally" — we traverse the array left to right, comparing elements, building maps. But the secret of this solution is to think vertically.

horizontal vs. vertical thinking ascii 
  HORIZONTAL (traditional): compare elements against each other
  ┌────┬────┬────┬────┬────┬────┬────┐
  │  5 │  5 │  7 │  5 │  9 │  9 │  9 │
  └────┴────┴────┴────┴────┴────┴────┘
    ↔    ↔    ↔    ↔    ↔    ↔
  "Is 5 == 5? Yes. Count[5]++. Is 5 == 7? No..."

  VERTICAL (bitwise): treat each BIT COLUMN as an independent counter
  ┌────┬────┬────┬────┬────┬────┬────┐
  │  5 │  5 │  7 │  5 │  9 │  9 │  9 │   ← decimal values (irrelevant)
  ├────┼────┼────┼────┼────┼────┼────┤
  │0101│0101│0111│0101│1001│1001│1001│   ← binary representation
  └────┴────┴────┴────┴────┴────┴────┘
    ↕    ↕    ↕    ↕    ↕    ↕    ↕
  bit3: 0  0  0  0  1  1  1  → count=3 → divisible by 3 → VANISH
  bit2: 1  1  1  1  0  0  0  → count=3 → divisible by 3 → VANISH
  bit1: 0  0  1  0  0  0  0  → count=1 → NOT divisible by 3 → SURVIVE (it's in 7!)
  bit0: 1  1  1  1  1  1  1  → count=7 → 7 mod 3 = 1 → SURVIVE (also in 7!)

  Result: bits 1 and 0 survived. 0b0011 = ... wait, 7 = 0b0111. Let me redo with bit2.
  7 = 0b0111, 5 = 0b0101, 9 = 0b1001

  bit3: 0+0+0+0+1+1+1 = 3 mod 3 = 0 → VANISH
  bit2: 1+1+1+1+0+0+0 = 4 mod 3 = 1 → SURVIVE  ← unique's bit
  bit1: 0+0+1+0+0+0+0 = 1 mod 3 = 1 → SURVIVE  ← unique's bit
  bit0: 1+1+1+1+1+1+1 = 7 mod 3 = 1 → SURVIVE  ← unique's bit

  Bits 2,1,0 survived → 0b0111 = 7 ✓

// A Analogia da Esteira Industrial

Imagine uma fábrica com 32 esteiras de seleção óptica funcionando em paralelo — uma para cada posição de bit. Cada produto que entra na fábrica ativa simultaneamente todas as esteiras correspondentes aos bits que ele tem ligados. Cada esteira tem dois silos: ones e twos.

Imagine a factory with 32 optical sorting conveyors running in parallel — one for each bit position. Each product entering the factory simultaneously activates all conveyors corresponding to its set bits. Each conveyor has two silos: ones and twos.

the factory with 32 parallel conveyors ascii 
  A product with value 5 (0b0101) arrives:

  Bit 31 ─── conveyor 31 ─── [ones_31=0] [twos_31=0]  (bit is 0: nothing happens)
  Bit 30 ─── conveyor 30 ─── [ones_30=0] [twos_30=0]  (bit is 0: nothing happens)
  ...
  Bit 2  ─── conveyor  2 ─── [ones_2 =?] [twos_2 =?]  (bit is 1: ACTIVATED!)
  Bit 1  ─── conveyor  1 ─── [ones_1 =0] [twos_1 =0]  (bit is 0: nothing happens)
  Bit 0  ─── conveyor  0 ─── [ones_0 =?] [twos_0 =?]  (bit is 1: ACTIVATED!)

  All 32 conveyors process in parallel in ONE CPU clock cycle.
  This is why it is O(1) space — we only need TWO 32-bit integers (ones, twos),
  each acting as 32 single-bit silos simultaneously.

A beleza física disso: quando dizemos ones = (ones ^ sig) & ~twos, não estamos executando isso sequencialmente para cada bit. O hardware do processador executa todas as 32 operações em paralelo, em um único ciclo de clock. Trinta e dois contadores independentes, processados ao mesmo tempo.

The physical beauty of this: when we say ones = (ones ^ sig) & ~twos, we are not executing this sequentially for each bit. The processor hardware runs all 32 operations in parallel, in a single clock cycle. Thirty-two independent counters, all processed at once.

03

As Portas de Hardware

Para construir nosso contador módulo-3, precisamos de exatamente duas ferramentas. Cada uma tem um papel físico muito específico.

To build our modulo-3 counter, we need exactly two tools. Each has a very specific physical role.

// XOR (^) — O Ejetor Toggle

XOR é o mecanismo de entrada e saída do silo. Ele tem uma propriedade perfeita para nosso propósito: se um token não está no silo (0), XOR com 1 coloca ele lá. Se já está lá (1), XOR com 1 o ejeta. É um interruptor toggle físico.

XOR is the silo's entry and exit mechanism. It has a perfect property for our purpose: if a token is not in the silo (0), XOR with 1 places it there. If it is already there (1), XOR with 1 ejects it. It is a physical toggle switch.

XOR — the Toggle Ejector ascii 
  silo_state ^ incoming_bit:

  0 ^ 1 = 1   ← silo was EMPTY, token ENTERS   (first appearance)
  1 ^ 1 = 0   ← silo was FULL,  token EJECTED  (second appearance: leaves ones, goes to twos)
  0 ^ 1 = 1   ← silo was EMPTY, token ENTERS   (but this is stopped by the Lock below!)

  Crucial: XOR alone would cause chaos.
  If unchecked, a bit would bounce back into 'ones' on its third pass.
  We need a LOCK.

// AND NOT (& ~) — A Trava Mecânica

& ~ é a trava que impede que um token entre no silo errado. ~twos significa "NOT twos" — uma máscara que é 0 em exatamente os bits onde twos é 1. Quando aplicamos & ~twos ao resultado do XOR, estamos dizendo: "qualquer bit que já está no silo twos não pode entrar no silo ones". É uma porta de exclusão física.

& ~ is the lock that prevents a token from entering the wrong silo. ~twos means "NOT twos" — a mask that is 0 at exactly the bits where twos is 1. When we apply & ~twos to the XOR result, we are saying: "any bit already in the twos silo cannot enter the ones silo". It is a physical exclusion gate.

AND NOT — the Mechanical Lock ascii 
  (ones ^ sig) & ~twos:

  If twos has a bit set (that bit has been seen TWICE already):
    ~twos has a 0 at that position
    AND with 0 = FORCED TO 0
    → entry to 'ones' is BLOCKED for that bit

  If twos does NOT have a bit (that bit has been seen 0 or 1 times):
    ~twos has a 1 at that position
    AND with 1 = PASSES THROUGH UNCHANGED
    → entry to 'ones' is ALLOWED for that bit

  The SAME logic applies symmetrically:
  (twos ^ sig) & ~ones: blocks entry to 'twos' if ones just captured it.

// Por que a ordem importa

As duas linhas do algoritmo DEVEM ser executadas nessa ordem. ones é calculado usando o twos antigo. Depois, twos é calculado usando o ones novo. Isso cria a interação correta: quando um bit entra em ones pela primeira vez, o twos atualizado usa esse novo estado de ones para se bloquear, garantindo que o bit não apareça nos dois silos ao mesmo tempo.

The two lines of the algorithm MUST execute in this order. ones is computed using the OLD twos. Then twos is computed using the NEW ones. This creates the correct interaction: when a bit first enters ones, the updated twos uses this new state of ones to block itself, ensuring the bit never appears in both silos at the same time.

04

A Obra-prima

Tudo que construímos culmina em duas linhas. Duas linhas que implementam 32 contadores paralelos módulo-3, em O(1) espaço, em O(1) tempo por elemento.

Everything we have built culminates in two lines. Two lines that implement 32 parallel modulo-3 counters, in O(1) space, in O(1) time per element.

find_unique.cpp — the complete solution cpp 
#include <vector>

int findUnique(const std::vector<int>& nums) {
    int ones = 0;   /* silo 1: bits seen (count % 3) == 1 */
    int twos = 0;   /* silo 2: bits seen (count % 3) == 2 */

    for (int sig : nums) {
        /*  ┌─────────────────────────────────────────────────────────────┐ */
        /*  │  THE MASTERPIECE: two lines, 32 parallel modulo-3 counters  │ */
        /*  └─────────────────────────────────────────────────────────────┘ */
        ones = (ones ^ sig) & ~twos;   /* XOR in, then lock out bits already in twos */
        twos = (twos ^ sig) & ~ones;   /* XOR in, then lock out bits already in ones */
    }

    /* After processing all elements:                           */
    /* Bits seen 3× → count%3==0 → in neither silo → 0         */
    /* Bits seen 1× → count%3==1 → still in 'ones' silo        */
    /* Answer lives in 'ones'.                                  */
    return ones;
}
state machine per bit position ascii 
  For each of the 32 independent bit positions:

  State 00   ──── see "1" ────▶   State 01   ──── see "1" ────▶   State 10
  ones=0                          ones=1                           ones=0
  twos=0                          twos=0                           twos=1
     ▲                                                                │
     └────────────────── see "1" ──────────────────────────────────────┘
                         (annihilated: count is now divisible by 3)

  "0" bit never activates (sig bit is 0 → no state change in that position)
  Only "1" bits drive the state transitions.

  After the 3rd "1":  State 00 again → completely gone from both silos.
  After the 1st "1":  State 01 → survives in 'ones' → this is the unique element.
05

Teste de Mesa: 3 Passes, 1 Annihilation

Este é o momento que vai fazer tudo encaixar. Vamos isolar exatamente uma posição de bit — o bit 0 — e rastrear o que acontece quando um número com esse bit ligado passa pelo algoritmo exatamente 3 vezes. Começamos do zero.

This is the moment that will make everything click. We will isolate exactly one bit position — bit 0 — and trace what happens when a number with that bit set passes through the algorithm exactly 3 times. Starting from zero.

initial state ascii 
  sig = 0b...0001  (any number with bit 0 = 1)
  ones = 0b...0000
  twos = 0b...0000

  SILOS (focusing on bit 0 only):
  ┌─────────────────────────────┐
  │  ONES silo:  bit 0 = [ 0 ] │  empty
  │  TWOS silo:  bit 0 = [ 0 ] │  empty
  └─────────────────────────────┘

// PASS 1 — First appearance: the bit ENTERS the ones silo

pass 1 — step by step (bit 0) ascii 
  sig (bit 0) = 1,   ones (bit 0) = 0,   twos (bit 0) = 0

  ── Computing new ones ──────────────────────────────────────────────────
  Step A: ones ^ sig   →   0 ^ 1 = 1    (XOR: silo was empty, token ENTERS)
  Step B: ~twos        →   ~0    = 1    (NOT twos: twos is 0, so lock is OPEN)
  Step C: result & lock →  1 & 1 = 1   ← new_ones = 1  ✓ token is in the silo!

  ── Computing new twos (using NEW ones = 1) ─────────────────────────────
  Step A: twos ^ sig   →   0 ^ 1 = 1    (XOR: silo was empty, token would enter)
  Step B: ~ones        →   ~1    = 0    (NOT new_ones: ones just captured it!)
  Step C: result & lock →  1 & 0 = 0   ← new_twos = 0  ✓ token is LOCKED OUT!

  ┌─────────────────────────────────────────┐
  │  ONES silo:  bit 0 = [ 1 ]  ← TOKEN IN │  count % 3 == 1 ✓
  │  TWOS silo:  bit 0 = [ 0 ]             │
  └─────────────────────────────────────────┘

// PASS 2 — Second appearance: ejected from ones, ENTERS twos

pass 2 — step by step (bit 0) ascii 
  sig (bit 0) = 1,   ones (bit 0) = 1,   twos (bit 0) = 0

  ── Computing new ones ──────────────────────────────────────────────────
  Step A: ones ^ sig   →   1 ^ 1 = 0    (XOR: silo was FULL, token EJECTED)
  Step B: ~twos        →   ~0    = 1    (NOT twos: twos is 0, lock is open)
  Step C: result & lock →  0 & 1 = 0   ← new_ones = 0  ✓ token EJECTED!

  ── Computing new twos (using NEW ones = 0) ─────────────────────────────
  Step A: twos ^ sig   →   0 ^ 1 = 1    (XOR: silo was empty, token enters)
  Step B: ~ones        →   ~0    = 1    (NOT new_ones: ones is now 0, lock is OPEN)
  Step C: result & lock →  1 & 1 = 1   ← new_twos = 1  ✓ token moves to twos!

  ┌──────────────────────────────────────────┐
  │  ONES silo:  bit 0 = [ 0 ]              │
  │  TWOS silo:  bit 0 = [ 1 ]  ← TOKEN IN │  count % 3 == 2 ✓
  └──────────────────────────────────────────┘

// PASS 3 — The Annihilation: blocked, ejected, EVAPORATED

pass 3 — the annihilation (bit 0) ascii 
  sig (bit 0) = 1,   ones (bit 0) = 0,   twos (bit 0) = 1

  ── Computing new ones ──────────────────────────────────────────────────
  Step A: ones ^ sig   →   0 ^ 1 = 1    (XOR: silo was empty, token would enter...)
  Step B: ~twos        →   ~1    = 0    (NOT twos: twos is FULL → LOCK IS CLOSED!)
  Step C: result & lock →  1 & 0 = 0   ← new_ones = 0  ✗ BLOCKED BY LOCK!

  ── Computing new twos (using NEW ones = 0) ─────────────────────────────
  Step A: twos ^ sig   →   1 ^ 1 = 0    (XOR: silo was FULL, token EJECTED)
  Step B: ~ones        →   ~0    = 1    (NOT new_ones: ones is 0, lock is open)
  Step C: result & lock →  0 & 1 = 0   ← new_twos = 0  ✓ token EJECTED!

  ┌─────────────────────────────────────────────┐
  │  ONES silo:  bit 0 = [ 0 ]                 │
  │  TWOS silo:  bit 0 = [ 0 ]                 │  count % 3 == 0 → ANNIHILATED ✓
  └─────────────────────────────────────────────┘

  The bit that appeared 3 times has completely EVAPORATED from both silos.
  It is as if it never existed.

// O que acabou de acontecer é físico e real

Nos três passes, o estado de dois bits (ones[0] e twos[0]) evoluiu: 00 → 01 → 10 → 00. Isso é um contador de 2 bits que conta módulo 3. Construímos isso com três operações bitwise. E esse contador está rodando em paralelo em 32 posições de bit simultaneamente, em dois inteiros de 32 bits, com custo de memória de exatamente 8 bytes. Total. Para um array de bilhões de elementos.

Over the three passes, the state of two bits (ones[0] and twos[0]) evolved: 00 → 01 → 10 → 00. That is a 2-bit counter counting modulo 3. We built it with three bitwise operations. And that counter is running in parallel across 32 bit positions simultaneously, in two 32-bit integers, with a memory cost of exactly 8 bytes. Total. For an array of billions of elements.

06

Prova Completa com Array Real

Agora vamos executar o algoritmo completo no array [2, 2, 3, 2] onde a resposta é 3. Vamos rastrear ones e twos a cada passo usando os valores binários reais.

Now let us run the full algorithm on the array [2, 2, 3, 2] where the answer is 3. We will trace ones and twos at every step using the actual binary values.

full trace of [2, 2, 3, 2] → answer: 3 ascii 
  Values: 2 = 0b010,   3 = 0b011
  Start:  ones = 0b000,  twos = 0b000

  ══ sig = 2 (0b010) ════════════════════════════════════════════════
  ones = (0b000 ^ 0b010) & ~0b000
       = (0b010)         &  0b111
       = 0b010                         ← bit 1 enters ones
  twos = (0b000 ^ 0b010) & ~0b010
       = (0b010)         &  0b101
       = 0b000                         ← locked out (ones has bit 1)
  State: ones = 010, twos = 000

  ══ sig = 2 (0b010) ════════════════════════════════════════════════
  ones = (0b010 ^ 0b010) & ~0b000
       = (0b000)         &  0b111
       = 0b000                         ← bit 1 ejected from ones
  twos = (0b000 ^ 0b010) & ~0b000
       = (0b010)         &  0b111
       = 0b010                         ← bit 1 enters twos
  State: ones = 000, twos = 010

  ══ sig = 3 (0b011) ════════════════════════════════════════════════
  ones = (0b000 ^ 0b011) & ~0b010
       = (0b011)         &  0b101
       = 0b001                         ← bit 0 enters ones (bit 1 blocked by twos)
  twos = (0b010 ^ 0b011) & ~0b001
       = (0b001)         &  0b110
       = 0b000                         ← bit 0 locked (ones has it), bit 1 ejected!
  State: ones = 001, twos = 000
  Note: 3's bit 0 was seen once → safely in ones.
        3's bit 1 was seen 3× total (twice for 2, once for 3) → ANNIHILATED.

  ══ sig = 2 (0b010) ════════════════════════════════════════════════
  ones = (0b001 ^ 0b010) & ~0b000
       = (0b011)         &  0b111
       = 0b011                         ← bit 1 enters ones (it's the 3rd 2-bit appearance!)
  twos = (0b000 ^ 0b010) & ~0b011
       = (0b010)         &  0b100
       = 0b000                         ← bit 1 blocked (ones has it)

  Wait — this doesn't look right yet. Let's verify carefully:
  2's bit 1 has appeared: pass1=entered ones, pass2=moved to twos, pass3=?

  ones = (ones ^ sig) & ~twos = (001 ^ 010) & ~000 = 011 & 111 = 011
  twos = (twos ^ sig) & ~ones = (000 ^ 010) & ~011 = 010 & 100 = 000

  State: ones = 011, twos = 000
  Hmm, bit 1 is in ones again? Let me recount.

  2's bit 1: appeared in sig=2 (3 times), appeared in sig=3 once → TOTAL = 4 times
  4 mod 3 = 1 → should be in ones! But 3's bit 1 = 1, 2's bit 1 = 1...

  Actually 3 = 0b011, so 3 contributes bit 1 as well.
  Bit 1 count: sig[0]=2→1, sig[1]=2→1, sig[2]=3→1, sig[3]=2→1 = 4 times
  4 mod 3 = 1 → bit 1 in ones ✓

  Bit 0 count: sig[2]=3→1 = 1 time total (only 3 has bit 0 set)
  1 mod 3 = 1 → bit 0 in ones ✓

  Final: ones = 0b011 = 3 ✓  THE ANSWER IS CORRECT.
verification — running the actual code cpp 
#include <iostream>
#include <vector>

int findUnique(const std::vector<int>& nums) {
    int ones = 0, twos = 0;
    for (int sig : nums) {
        ones = (ones ^ sig) & ~twos;
        twos = (twos ^ sig) & ~ones;
    }
    return ones;
}

int main() {
    std::cout << findUnique({2, 2, 3, 2})           << "\n"; // 3
    std::cout << findUnique({5, 5, 7, 5, 9, 9, 9})  << "\n"; // 7
    std::cout << findUnique({0, 1, 0, 1, 0, 1, 99}) << "\n"; // 99
    return 0;
}

/* Output:
   3
   7
   99
*/

/* Time:  O(n) — single pass through the array           */
/* Space: O(1) — exactly two int variables, always       */
/* Works for arrays of any size, from 3 to 3 billion     */
