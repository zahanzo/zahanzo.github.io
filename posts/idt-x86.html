01

Polling vs. Interrupções

Polling significa o CPU verificar ativamente e repetidamente um registrador de status de dispositivo para detectar um evento. É determinístico e simples, mas desperdiça 100% dos ciclos do núcleo enquanto aguarda — inaceitável para um sistema com dezenas de dispositivos. Interrupções invertem o fluxo: o dispositivo sinaliza o CPU quando tem dados, e o CPU responde sob demanda.

Polling means the CPU actively and repeatedly checks a device status register to detect an event. It is deterministic and simple, but wastes 100% of the core's cycles while waiting — unacceptable for a system with dozens of devices. Interrupts invert the flow: the device signals the CPU when it has data, and the CPU responds on demand.

// Três categorias de eventos

taxonomia de interrupções e exceções no x86_64 ascii 
  HARDWARE INTERRUPTS  (IRQ — maskable via RFLAGS.IF=0)
  ├── Vectors 32–255  — assigned by OS/BIOS/APIC
  ├── Source: timer, keyboard, NIC, disk controller → via PIC or APIC
  ├── Asynchronous: can arrive between ANY two instructions
  └── CPU must send EOI (End Of Interrupt) to PIC/APIC before re-enabling

  EXCEPTIONS  (generated by the CPU itself — synchronous)
  ├── Faults    RIP saved = faulting instruction (restartable after fix)
  │   └── #PF (14), #GP (13), #SS (12), #NP (11), #TS (10), #DE (0)
  ├── Traps     RIP saved = NEXT instruction (non-restartable)
  │   └── #BP (3) INT3, #OF (4), #DB (1) single-step
  └── Aborts    RIP may be invalid — non-recoverable
      └── #DF (8) double fault, #MC (18) machine check

  SOFTWARE INTERRUPTS  (explicit instruction)
  ├── INT n     — triggers vector n (legacy, slow, e.g. INT 0x80 for Linux syscall)
  ├── SYSCALL   — fast path to ring 0 (no IDT lookup, uses MSR_LSTAR)
  └── INT3      — single-byte breakpoint (0xCC), used by debuggers

Por que SYSCALL não usa a IDT?

INT 0x80 executa um lookup completo na IDT, troca de privilégio via TSS e empurra o frame de interrupção completo — ~100+ ciclos. A instrução SYSCALL (introduzida no AMD K6, mandatória em x86_64) usa um caminho otimizado: o endereço do kernel handler é carregado do MSR_LSTAR, o RIP do usuário é salvo em RCX, e RFLAGS em R11. Não há stack switch via TSS, não há push automático de frame — o handler precisa fazer isso manualmente. É ~3× mais rápido, mas exige que o kernel implemente o protocolo correto (como faz o Linux em entry_SYSCALL_64).

INT 0x80 performs a full IDT lookup, privilege switch via TSS, and pushes the complete interrupt frame — ~100+ cycles. The SYSCALL instruction (introduced in AMD K6, mandatory in x86_64) uses an optimized path: the kernel handler address is loaded from MSR_LSTAR, the user RIP is saved in RCX, and RFLAGS in R11. There is no TSS stack switch, no automatic frame push — the handler must do this manually. It is ~3× faster, but requires the kernel to implement the correct protocol (as Linux does in entry_SYSCALL_64).

02

Anatomia da IDT

A IDT (Interrupt Descriptor Table) é um array de até 256 entradas de 16 bytes cada, localizado em qualquer endereço físico. O registro IDTR (carregado com LIDT) aponta para ela: 16 bits de limite + 64 bits de base. Cada entrada é um Gate Descriptor que descreve como chegar ao handler da interrupção e com qual privilégio.

The IDT (Interrupt Descriptor Table) is an array of up to 256 entries of 16 bytes each, located at any physical address. The IDTR register (loaded with LIDT) points to it: 16-bit limit + 64-bit base. Each entry is a Gate Descriptor that describes how to reach the interrupt handler and with what privilege.

estrutura de um Gate Descriptor de 16 bytes (64-bit mode) ascii 
  Bytes  0–1   Offset[15:0]   — low 16 bits of ISR address
  Bytes  2–3   Selector       — code segment selector (usually kernel CS = 0x08)
  Byte   4     IST[2:0]       — Interrupt Stack Table index (0 = use current stack)
               Reserved[4:3]
  Byte   5     Gate Type[3:0] — 0xE = Interrupt Gate | 0xF = Trap Gate
               Zero[4]
               DPL[6:5]       — Descriptor Privilege Level (0=ring0 only, 3=user allowed)
               Present[7]     — must be 1 for valid entry
  Bytes  6–7   Offset[31:16]  — middle 16 bits of ISR address
  Bytes  8–11  Offset[63:32]  — high 32 bits of ISR address
  Bytes 12–15  Reserved       — must be zero

  ┌──────────────────────────────────────────────────────────────┐
  │  ISR full address = Offset[63:32] : Offset[31:16] : Offset[15:0]  │
  └──────────────────────────────────────────────────────────────┘

// Interrupt Gate vs Trap Gate — a diferença crítica

// INTERRUPT GATE (Type 0xE)
Limpa RFLAGS.IF — interrupções desabilitadas
Previne preempção durante execução
Uso: IRQs de hardware (timer, teclado, NIC)
ISR lenta = latência do sistema toda presa
// TRAP GATE (Type 0xF)
NÃO altera IF — interrupções habilitadas
IRQs podem preemptar o handler
Uso: breakpoints (#BP), single-step (#DB)
Em IRQ de hardware = nesting infinito possível

// IST — Interrupt Stack Table e o TSS

O campo IST (3 bits) do Gate Descriptor referencia um dos 7 stacks alternativos definidos no TSS (Task State Segment). Quando uma exceção com IST≠0 ocorre, a CPU incondicionalmente troca para aquele stack — mesmo que RSP já esteja corrompido. Isso é essencial para exceções críticas que podem ocorrer com o stack do kernel em estado inválido:

The IST (3-bit) field in the Gate Descriptor references one of 7 alternate stacks defined in the TSS (Task State Segment). When an exception with IST≠0 occurs, the CPU unconditionally switches to that stack — even if RSP is already corrupted. This is essential for critical exceptions that may occur with the kernel stack in an invalid state:

uso típico do IST no Linux/kernel hobby ascii 
  TSS.IST1  →  #NMI  (Non-Maskable Interrupt)     — can interrupt ANY context
  TSS.IST2  →  #DF   (Double Fault, vector 8)      — triggered when handler itself faults
  TSS.IST3  →  #MC   (Machine Check, vector 18)    — hardware error, RSP might be invalid
  TSS.IST4  →  #DB   (Debug, vector 1)             — single-step, JTAG

  IST = 0   →  use the stack at TSS.RSP0 (kernel stack of current task)
               — standard for all other interrupts and exceptions

O que acontece sem IST para o Double Fault?

Se um handler de exceção (ex: #GP) falha — por exemplo, porque o stack do kernel está corrompido e RSP aponta para memória inválida — o CPU tenta invocar o handler de #DF. Sem IST, ele tentaria usar o mesmo stack corrompido para salvar o frame de #DF. Isso gera uma terceira exceção enquanto tratando outra → triple fault → o processador reseta a máquina imediatamente, sem nenhum aviso. Com IST, o #DF sempre usa um stack separado e garantidamente válido.

If an exception handler (e.g., #GP) faults — for example because the kernel stack is corrupted and RSP points to invalid memory — the CPU attempts to invoke the #DF handler. Without IST, it would try to use the same corrupted stack to save the #DF frame. This triggers a third exception while handling another → triple fault → the processor immediately resets the machine without any warning. With IST, #DF always uses a separate, guaranteed-valid stack.

03

O Fluxo Atômico

No instante em que uma interrupção é reconhecida pela CPU, uma sequência precisa e atômica de passos de hardware ocorre — sem software envolvido. Entender essa sequência é pré-requisito para escrever qualquer ISR correta.

The instant an interrupt is acknowledged by the CPU, a precise and atomic sequence of hardware steps occurs — no software involved. Understanding this sequence is a prerequisite for writing any correct ISR.

sequência de hardware ao receber interrupção (x86_64) ascii 
  1. CPU finishes (or abandons) the current instruction.
  2. Checks RFLAGS.IF for maskable IRQs (NMI ignores this).

  ── IF PRIVILEGE CHANGE (user ring3 → kernel ring0) ────────────────────
  3a. Load new RSP from TSS.RSP0 (or TSS.IST[n] if IST ≠ 0)     ← stack switch!
  3b. Switch to new SS (kernel data segment)
  3c. On the NEW kernel stack, push:
      ┌──────────────────────────┐  ← new RSP after all pushes
      │  SS        (old, ring3)  │  +40  user stack segment
      │  RSP       (old, ring3)  │  +32  user stack pointer
      │  RFLAGS    (old)         │  +24  saved flags
      │  CS        (old, ring3)  │  +16  user code segment
      │  RIP       (old)         │  + 8  faulting / next instruction
      │  Error Code (if any)     │  + 0  ← pushed ONLY by: #DF #TS #NP #SS #GP #PF #AC #CP
      └──────────────────────────┘  ← RSP now

  ── IF NO PRIVILEGE CHANGE (kernel → kernel, already ring0) ────────────
  3d. Use SAME stack. Push only:
      ┌──────────────────────────┐
      │  RFLAGS    (old)         │  +16   (SS and RSP are NOT saved)
      │  CS        (old)         │  + 8
      │  RIP       (old)         │  + 0
      │  Error Code (if any)     │
      └──────────────────────────┘

  4. If Interrupt Gate: clear RFLAGS.IF  (disable further maskable IRQs)
  5. Load CS and RIP from the Gate Descriptor (jump to ISR)
  6. Software takes over (your ISR runs)

// Quais exceções empurram error code?

vetores que empurram error code automaticamente ascii 
  Vector  Mnemonic  Name                       Error Code?
  ──────  ────────  ─────────────────────────  ──────────
    8       #DF     Double Fault               YES (always 0)
   10       #TS     Invalid TSS                YES
   11       #NP     Segment Not Present        YES
   12       #SS     Stack-Segment Fault        YES
   13       #GP     General Protection Fault   YES
   14       #PF     Page Fault                 YES  (see previous post for bit layout)
   17       #AC     Alignment Check            YES (always 0)
   21       #CP     Control Protection Fault   YES

  All others (0–7, 9, 15–16, 18–20, 22–31, 32–255): NO error code.
  ; ISR stubs must handle this asymmetry — see Section 04.

Segurança: interrupt nesting e stack overflow do kernel

Cada frame de interrupção ocupa 40 bytes no stack (sem error code, sem privilégio, kernel→kernel). O stack padrão do kernel no Linux é 16 KB. Uma ISR mal escrita que: (1) usa Trap Gate para IRQ periódica, (2) demora mais que o período do timer, e (3) não reenvia EOI — pode acumular centenas de frames antes do stack overflow. O resultado é um #DF silencioso que, sem IST configurado, produz um triple fault e reset da máquina. O diagnóstico é quase impossível sem um debugger de kernel (KDB/KGDB) ou memória de dump post-mortem.

Each interrupt frame occupies 40 bytes on the stack (no error code, no privilege change, kernel→kernel). The Linux default kernel stack is 16 KB. A badly written ISR that: (1) uses a Trap Gate for a periodic IRQ, (2) runs longer than the timer period, and (3) does not send EOI — can accumulate hundreds of frames before a stack overflow. The result is a silent #DF which, without IST configured, produces a triple fault and machine reset. The diagnosis is nearly impossible without a kernel debugger (KDB/KGDB) or a post-mortem memory dump.

Implementação da ISR: Assembly + C + IRETQ

O CPU salva apenas o frame mínimo (RIP, CS, RFLAGS, RSP, SS). O restante dos registradores — RAX a R15 — pode ter sido modificado pelo contexto interrompido e deve ser salvo pelo software. O padrão é um stub em Assembly que: (1) normaliza o stack com um error code fictício para exceções que não o empurram, (2) salva todos os registradores, (3) chama um handler em C, e (4) restaura tudo antes de IRETQ.

The CPU saves only the minimum frame (RIP, CS, RFLAGS, RSP, SS). The remaining registers — RAX through R15 — may have been modified by the interrupted context and must be saved by software. The standard pattern is an Assembly stub that: (1) normalizes the stack with a fake error code for exceptions that don't push one, (2) saves all registers, (3) calls a C handler, and (4) restores everything before IRETQ.

isr_stubs.asm — NASM, 64-bit nasm 
; Macro: generate a stub for exceptions WITHOUT error code
; We push a fake 0 to keep the stack layout uniform.
%macro ISR_NOERR 1
global isr%1
isr%1:
    push  qword 0          ; fake error code
    push  qword %1         ; vector number
    jmp   isr_common
%endmacro

; Macro: generate a stub for exceptions WITH error code
; CPU already pushed it; we only push the vector number.
%macro ISR_ERR 1
global isr%1
isr%1:
    push  qword %1         ; vector number (error code already on stack)
    jmp   isr_common
%endmacro

; --- Generate all 256 stubs ---
ISR_NOERR  0   ; #DE  Divide Error
ISR_NOERR  1   ; #DB  Debug
ISR_NOERR  2   ; NMI
ISR_NOERR  3   ; #BP  Breakpoint
ISR_NOERR  6   ; #UD  Invalid Opcode
ISR_ERR    8   ; #DF  Double Fault     (error code = 0, always)
ISR_ERR   13   ; #GP  General Protection
ISR_ERR   14   ; #PF  Page Fault
ISR_NOERR 32   ; IRQ0 timer
ISR_NOERR 33   ; IRQ1 keyboard
; ... (continue for all 256 vectors)

section .text

isr_common:
    ; At entry, stack (top to bottom):
    ;   [rsp+0]  = vector,      [rsp+8]  = error code (real or fake)
    ;   [rsp+16] = RIP,         [rsp+24] = CS
    ;   [rsp+32] = RFLAGS,      [rsp+40] = RSP (if ring3→ring0)
    ;   [rsp+48] = SS           (if ring3→ring0)

    ; Save all general-purpose registers (caller + callee saved)
    push  rax
    push  rbx
    push  rcx
    push  rdx
    push  rsi
    push  rdi
    push  rbp
    push  r8
    push  r9
    push  r10
    push  r11
    push  r12
    push  r13
    push  r14
    push  r15

    mov   rdi, rsp          ; 1st arg: pointer to interrupt_frame (System V ABI)
    call  isr_dispatch     ; call C handler

    ; Restore all GP registers
    pop   r15
    pop   r14
    pop   r13
    pop   r12
    pop   r11
    pop   r10
    pop   r9
    pop   r8
    pop   rbp
    pop   rdi
    pop   rsi
    pop   rdx
    pop   rcx
    pop   rbx
    pop   rax

    add   rsp, 16           ; discard vector number + error code
    iretq                   ; restore RIP, CS, RFLAGS, RSP, SS
isr_dispatch.c — C handler with full context access c 
#include <stdint.h>

/* Matches the exact layout pushed by isr_common + CPU frame */
typedef struct {
    /* pushed by isr_common (in reverse push order) */
    uint64_t r15, r14, r13, r12, r11, r10, r9, r8;
    uint64_t rbp, rdi, rsi, rdx, rcx, rbx, rax;
    /* pushed by ISR stub */
    uint64_t vector;
    uint64_t error_code;     /* real or fake (0) */
    /* pushed by CPU hardware */
    uint64_t rip;
    uint64_t cs;
    uint64_t rflags;
    uint64_t rsp;            /* only valid if ring3 → ring0 */
    uint64_t ss;             /* only valid if ring3 → ring0 */
} interrupt_frame_t;

/* Called from assembly stub — pointer to full saved context */
void isr_dispatch(interrupt_frame_t *f) {
    switch (f->vector) {
        case  3: handle_breakpoint(f);         break;
        case 13: handle_general_protection(f);  break;
        case 14: handle_page_fault(f);          break;
        case 32: handle_timer_irq(f);           break;
        default: handle_unknown(f);             break;
    }
}

void handle_page_fault(interrupt_frame_t *f) {
    uint64_t fault_addr;
    __asm__ volatile("mov %%cr2, %0" : "=r"(fault_addr));
    /* error_code bits: P=0 not present, W=1 write, U=1 user mode */
    /* dispatch to demand paging, CoW, or SIGSEGV as appropriate   */
    paging_handle_fault(fault_addr, f->error_code, f->cs & 3);
    /* no EOI needed for exceptions — only for hardware IRQs */
}

IRETQ — o oposto atômico do fluxo de entrada

IRETQ executa atomicamente: lê RIP, CS, RFLAGS do stack; se CS indica mudança de privilégio para ring 3, também restaura RSP e SS. Restaura RFLAGS.IF a partir do valor salvo — por isso o handler não precisa chamar STI explicitamente. Se o frame na stack estiver corrompido (wrong CS, RSP inválido), IRETQ dispara um #GP → potencial triple fault. É o ponto mais sensível de toda a ISR.

IRETQ executes atomically: reads RIP, CS, RFLAGS from the stack; if CS indicates a privilege change back to ring 3, it also restores RSP and SS. It restores RFLAGS.IF from the saved value — which is why the handler does not need an explicit STI. If the frame on the stack is corrupted (wrong CS, invalid RSP), IRETQ triggers a #GP → potential triple fault. It is the single most sensitive point of the entire ISR.

// Prioridades no APIC: TPR e a classe do vetor

No APIC, a prioridade de um vetor é definida pelos seus 4 bits superiores (priority class = vector[7:4]). Vetores 0xF0–0xFF têm prioridade 15 (máxima), 0x20–0x2F têm prioridade 2 (mínima usável). O TPR (Task Priority Register, CR8 em x86_64) define o limiar mínimo de prioridade que o Local APIC aceita: interrupções com prioridade ≤ TPR são adiadas até que TPR seja reduzido.

In the APIC, the priority of a vector is defined by its upper 4 bits (priority class = vector[7:4]). Vectors 0xF0–0xFF have priority 15 (maximum), 0x20–0x2F have priority 2 (minimum usable). The TPR (Task Priority Register, CR8 in x86_64) defines the minimum priority threshold the Local APIC accepts: interrupts with priority ≤ TPR are deferred until TPR is lowered.

estratégia de prioridade para kernel e embedded ascii 
  Vector range    Priority class   Typical assignment
  ────────────────────────────────────────────────────────────────
  0x00–0x1F       0–1              CPU exceptions (non-configurable)
  0x20–0x2F       2                Low-priority IRQs (disk, sound)
  0x30–0x9F       3–9              General device IRQs (NIC, USB)
  0xA0–0xCF       10–12            High-priority devices
  0xD0–0xDF       13               Scheduler / IPI (Linux: RESCHEDULE_VECTOR)
  0xE0–0xEF       14               TLB shootdown, call function IPIs
  0xF0–0xFE       15               Thermal, error, NMI-like
  0xFF            15               APIC spurious vector (must always be registered)

  ; Embedded pattern (RTOS on x86): assign higher vectors to time-critical ISRs,
  ; set CR8 (TPR) to mask lower-priority IRQs inside critical sections.
  ; Equivalent to BASEPRI register on ARM Cortex-M.

// Top Half / Bottom Half — o modelo de ISR em dois estágios

Uma ISR que executa trabalho pesado com interrupções desabilitadas aumenta a interrupt latency do sistema inteiro. A solução universal é dividir o trabalho:

An ISR that performs heavy work with interrupts disabled increases the interrupt latency of the entire system. The universal solution is to split the work:

// TOP HALF (ISR)
Executa com IF=0, latência mínima
Lê dados do dispositivo para buffer circular
Envia EOI ao APIC
Agenda o Bottom Half
Proibido: sleep, malloc, mutex, I/O
// BOTTOM HALF (deferred)
Executa em contexto preemptível
Processa dados do buffer (decodificação, demux)
Pode dormir, alocar memória, adquirir mutexes
Linux: softirq/tasklet/workqueue
Windows: DPC em DISPATCH_LEVEL
exemplo: ISR de timer seguindo o modelo Top/Bottom Half c 
/* --- TOP HALF: runs inside ISR with interrupts disabled --- */
void handle_timer_irq(interrupt_frame_t *f) {
    /* 1. Read hardware counter (fast) */
    uint64_t ticks = lapic_read_timer_count();

    /* 2. Update kernel tick counter (atomic, no lock needed — IRQs off) */
    jiffies++;

    /* 3. Signal the scheduler to run (bottom half) */
    need_resched = 1;

    /* 4. Send EOI to Local APIC — BEFORE re-enabling interrupts */
    lapic_write(LAPIC_EOI, 0);

    /* Handler returns → IRETQ restores RFLAGS with IF=1 (for Interrupt Gate) */
}

/* --- BOTTOM HALF: runs in the scheduler context, preemptible --- */
void scheduler_tick(void) {        /* called by the kernel main loop */
    if (!need_resched) return;
    need_resched = 0;

    /* Heavy work: update process timeslices, select next task, etc. */
    sched_update_timeslice(current_task());
    sched_maybe_preempt();     /* may call schedule() and context-switch */
}

EOI: antes ou depois de reabilitar interrupções?

O EOI deve ser enviado ao APIC antes de sair da ISR. Se enviado depois do IRETQ (já com IF=1), o hardware pode reenviar a mesma interrupção antes que o EOI seja processado — causando um spurious interrupt ou, pior, nesting acidental. A ordem correta é sempre: (1) fazer o trabalho mínimo, (2) enviar EOI, (3) retornar via IRETQ. O APIC tem um registrador de "In-Service" (ISR do APIC) que mantém a interrupção como ativa até o EOI — entregar o EOI late pode causar deadlock na linha de prioridade.

The EOI must be sent to the APIC before exiting the ISR. If sent after IRETQ (with IF=1 already), the hardware may re-fire the same interrupt before the EOI is processed — causing a spurious interrupt or, worse, accidental nesting. The correct order is always: (1) do minimum work, (2) send EOI, (3) return via IRETQ. The APIC has an In-Service Register (APIC ISR) that keeps the interrupt marked as active until EOI — late EOI delivery can deadlock the priority line.
