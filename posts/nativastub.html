01

O problema: por que não confiar na ntdll.dll?

Todo programa Windows que faz algo útil — abrir arquivo, alocar memória, criar processo — eventualmente passa pela ntdll.dll. Ela é a única DLL carregada diretamente pelo kernel e funciona como a ponte entre o espaço do usuário e o NT Kernel. Ferramentas de segurança como EDRs sabem exatamente disso, e é exatamente ali que elas instalam seus hooks.

Every Windows program that does anything useful — open a file, allocate memory, create a process — eventually goes through ntdll.dll. It is the only DLL loaded directly by the kernel, acting as the bridge between user space and the NT Kernel. Security tools like EDRs know this, and that is exactly where they install their hooks.

// FLUXO NORMAL
programa.exe
kernel32 / ucrtbase
ntdll.dll ← hook do EDR
instrução syscall
NT Kernel
// DIRECT SYSCALLS
programa.exe
stub .asm (nosso código)
instrução syscall
NT Kernel

A técnica de Direct Syscalls elimina a ntdll.dll do caminho chamando a instrução syscall diretamente com o número correto da função no kernel (o SSN — Syscall Service Number). Nenhum hook instalado na ntdll.dll é alcançado.

Direct Syscalls eliminates ntdll.dll from the call path by invoking the syscall instruction directly with the kernel's function number (the SSN — Syscall Service Number). No hook installed in ntdll.dll is ever reached.

02

O NativaStub Gen

O NativaStub Gen é um gerador de código escrito em C++. Ele recebe a ntdll.dll como entrada, faz análise estática das exportações, extrai o SSN de cada função Nt*, e gera Assembly + header C prontos para compilar.

NativaStub Gen is a code generator written in C++. It takes ntdll.dll as input, statically analyzes its exports, extracts the SSN from each Nt* function, and outputs ready-to-compile Assembly + C header code.

// Como o SSN é extraído

Dentro de cada função Nt* exportada pela ntdll.dll, a Microsoft segue sempre o mesmo padrão de stub. O gerador varre cada exportação, localiza a instrução mov eax, N na sequência de bytes esperada e extrai o valor N — que é o índice da syscall na SSDT (System Service Descriptor Table) do kernel.

Inside each Nt* function exported by ntdll.dll, Microsoft always follows the same stub pattern. The generator scans each export, locates the mov eax, N instruction in the expected byte sequence, and extracts the value N — which is the syscall's index in the kernel's SSDT (System Service Descriptor Table).

ntdll.dll — NtCreateFile (desmontado) asm 
NtCreateFile:
    mov    r10, rcx
    mov    eax, 0x55           ; <-- SSN extracted by NativaStub Gen
    test   byte ptr [0x7FFE0308], 1
    jne    +0xF
    syscall
    ret

A migração: MASM + C++ → NASM + C

A versão original do NativaStub Gen foi desenvolvida no Visual Studio, usando MASM para o Assembly e gerando um .hpp que incluía <windows.h>. Isso funcionava, mas criava uma dependência implícita: qualquer projeto que usasse o header gerado precisaria do ambiente Win32 completo.

The original version of NativaStub Gen was developed in Visual Studio, using MASM for Assembly and generating a .hpp that included <windows.h>. This worked, but created an implicit dependency: any project using the generated header would require the full Win32 environment.

Para o proof-of-concept que queria construir — um executável nativo sem nenhuma importação — isso era um problema. A solução foi modificar as duas funções de geração do projeto.

For the proof-of-concept I wanted to build — a native executable with zero imports — this was a blocker. The solution was to modify the two generation functions of the project.

// GenerateAsm() — de MASM para NASM

— ANTES (MASM)
// MASM section directive
asmFile << ".code\n\n";

for (size_t i = 0; ...) {
    // proc/endp: MASM-only syntax
    asmFile << syscalls[i].name
            << " proc\n";
    asmFile << "    mov r10, rcx\n";
    asmFile << "    mov eax, " << i ...;
    asmFile << "    syscall\n";
    asmFile << "    ret\n";
    asmFile << syscalls[i].name
            << " endp\n\n";
}
// required MASM terminator
asmFile << "end\n";
+ DEPOIS (NASM)
// NASM standard section
asmFile << "section .text\n\n";

for (size_t i = 0; ...) {
    // exposes symbol to the linker
    asmFile << "global "
            << syscalls[i].name << "\n";
    // NASM label ends with ":"
    asmFile << syscalls[i].name
            << ":\n";
    asmFile << "    mov r10, rcx\n";
    asmFile << "    mov eax, " << i ...;
    asmFile << "    syscall\n";
    asmFile << "    ret\n\n";
}
// no "end" needed in NASM

Por que essa diferença importa?

No MASM, proc/endp são diretivas do assembler que definem um procedimento e automaticamente o exportam para o linker. No NASM, o mecanismo é explícito: global NomeDaFunção declara o símbolo como público, e o label NomeDaFunção: marca onde o código começa. Mais verboso, porém totalmente portátil — funciona com MinGW, Clang e qualquer toolchain que aceite NASM.

In MASM, proc/endp are assembler directives that define a procedure and automatically export it to the linker. In NASM, the mechanism is explicit: global FunctionName declares the symbol as public, and the label FunctionName: marks where the code starts. More verbose, but fully portable — works with MinGW, Clang, and any toolchain that accepts NASM.

// GenerateHpp() → GenerateHeader() — removendo <windows.h>

— ANTES (.hpp + windows.h)
hppFile << "#pragma once\n";
// pulls the entire Win32 API
hppFile << "#include <windows.h>\n\n";

// extern "C": disables C++ name mangling
// without it the linker looks for
// "?NtCreateFile@@YAJXZ" and fails
hppFile << "extern \"C\" {\n";

for (const auto& sc : syscalls) {
    hppFile << "    NTSTATUS "
            << sc.name
            << "(...);\n";
}
hppFile << "}\n";
+ DEPOIS (.h sem dependências)
hFile << "#pragma once\n\n";

// minimal typedefs — no windows.h
hFile << "typedef void* HANDLE;\n";
hFile << "typedef long NTSTATUS;\n";
hFile << "typedef void* PVOID;\n\n";

// pure C: no extern "C" needed,
// C never applies name mangling
for (const auto& sc : syscalls) {
    hFile << "NTSTATUS "
          << sc.name
          << "(...);\n";
}

O problema do extern "C"

O C++ aplica name mangling em todas as funções — um mecanismo que codifica os tipos dos parâmetros no nome do símbolo para suportar sobrecarga. Sem extern "C", o linker procuraria algo como ?NtCreateFile@@YAJXZ em vez de NtCreateFile, e a linkagem falharia silenciosamente. Ao migrar para C puro, o problema desaparece: C não tem name mangling.

C++ applies name mangling to all functions — a mechanism that encodes parameter types into the symbol name to support overloading. Without extern "C", the linker would look for something like ?NtCreateFile@@YAJXZ instead of NtCreateFile, and linking would silently fail. By migrating to pure C, the problem disappears: C has no name mangling.

Prova de conceito: zero importações

Com o syscalls.h e o syscalls.asm gerados, construí uma aplicação que roda no Subsistema Nativo do Windows — sem Win32, sem CRT, sem nenhum #include além do header gerado. O programa cria um arquivo em C:\ e escreve uma string nele usando apenas Direct Syscalls.

With the generated syscalls.h and syscalls.asm, I built an application that runs on the Windows Native Subsystem — no Win32, no CRT, no #include beyond the generated header. The program creates a file in C:\ and writes a string to it using only Direct Syscalls.

main.c c 
#include "syscalls.h"   /* only include — auto-generated by NativaStub Gen */

typedef unsigned short     USHORT;
typedef unsigned long      ULONG;
typedef unsigned long long ULONG_PTR;

typedef struct {
    USHORT Length, MaximumLength; USHORT* Buffer;
} UNICODE_STRING;

typedef struct {
    ULONG  Length;  HANDLE RootDirectory;
    UNICODE_STRING* ObjectName;  ULONG Attributes;
    PVOID  SecurityDescriptor, SecurityQualityOfService;
} OBJECT_ATTRIBUTES;

typedef struct {
    union { long Status; void* Pointer; };
    ULONG_PTR Information;
} IO_STATUS_BLOCK;

/* native subsystem entry point — not main() */
void EntryPoint() {
    HANDLE hFile;  IO_STATUS_BLOCK iosb;
    OBJECT_ATTRIBUTES oa;  UNICODE_STRING fileName;

    /* path in NT Object Manager format — \??\ maps to the device namespace */
    unsigned short path[] = {
        '\\','?','?','\\','C',':','\\',
        'N','o','S','T','D','_',
        'N','a','t','i','v','e','.','t','x','t', 0
    };

    fileName.Buffer = path;
    fileName.Length = 23 * 2;
    fileName.MaximumLength = 24 * 2;

    oa.Length = sizeof(OBJECT_ATTRIBUTES);
    oa.RootDirectory = 0;
    oa.ObjectName = &fileName;
    oa.Attributes = 0x40;             /* OBJ_CASE_INSENSITIVE */
    oa.SecurityDescriptor = oa.SecurityQualityOfService = 0;

    NTSTATUS st = NtCreateFile(
        &hFile,
        0x40000000 | 0x00100000,        /* GENERIC_WRITE | SYNCHRONIZE */
        &oa, &iosb,
        0, 0x80,                         /* no alloc size, FILE_ATTRIBUTE_NORMAL */
        0x01,                            /* FILE_SHARE_READ */
        5,                               /* FILE_OVERWRITE_IF */
        0x20 | 0x40,                     /* SYNC_IO_NONALERT | NON_DIRECTORY_FILE */
        0, 0
    );

    if (st == 0) {
        char msg[] = "Success via Direct Syscalls — no ntdll, no imports.\n";
        NtWriteFile(hFile, 0, 0, 0, &iosb, msg, sizeof(msg) - 1, 0, 0);
        NtClose(hFile);
    }

    /* terminate passing the NT status as exit code */
    NtTerminateProcess((HANDLE)-1, st);
}

Detalhes técnicos

// Por que mov r10, rcx?

Na ABI x64 do Windows, o primeiro argumento é passado em RCX. Mas a instrução syscall usa RCX para salvar o endereço de retorno (RIP) antes de entrar no kernel. O kernel então espera o primeiro argumento em R10 — daí o mov r10, rcx que abre cada stub.

In the Windows x64 ABI, the first argument is passed in RCX. But the syscall instruction uses RCX to save the return address (RIP) before entering the kernel. The kernel then expects the first argument in R10 — hence the mov r10, rcx that opens each stub.

// SSNs mudam entre versões do Windows

Os números de syscall não são estáveis — a Microsoft reordena a SSDT (System Service Descriptor Table) a cada grande atualização ou build do Windows. Em vez de adotar uma abordagem estática ou engessada com números hardcoded (como assumir que NtCreateFile é sempre 0x55), o NativaStub Gen resolve esse problema mapeando a arquitetura em disco.

A lógica explora uma característica do compilador da Microsoft: as funções Nt*/Zw* são dispostas na seção .text da ntdll.dll sequencialmente de acordo com o seu endereço de memória. A função com o menor RVA (Relative Virtual Address) corresponde ao SSN 0x00, a segunda ao 0x01, e assim por diante.

O parser em C++ (PEParser.cpp) lê a Export Directory, filtra os símbolos, e aplica um std::sort baseado nos RVAs. Com isso, os stubs em Assembly são gerados dinamicamente com os índices perfeitamente sincronizados para a versão exata do sistema operacional onde a ferramenta foi executada.

Syscall service numbers are inherently unstable — Microsoft reshuffles the SSDT (System Service Descriptor Table) with every major Windows build or update. Instead of relying on a fragile static approach with hardcoded values (such as assuming NtCreateFile is always 0x55), NativaStub Gen solves this problem by mapping the disk architecture.

The logic exploits a behavior of Microsoft's compiler: Nt*/Zw* functions are laid out within the .text section of ntdll.dll sequentially based on their memory addresses. The function with the lowest RVA (Relative Virtual Address) maps to SSN 0x00, the next to 0x01, and so on.

The C++ parser (PEParser.cpp) reads the Export Directory, filters the symbols, and applies a std::sort based on their RVAs. Consequently, the Assembly stubs are generated dynamically with indexes perfectly synchronized for the exact OS build where the tool ran.

// O path \??\C:\

O NT Object Manager não entende caminhos Win32 como C:\arquivo.txt. O prefixo \??\ é um link simbólico dentro do namespace do Object Manager que aponta para o namespace de dispositivos da sessão atual — equivalente ao \\.\ do Win32, mas acessível apenas via API nativa.

The NT Object Manager does not understand Win32 paths like C:\file.txt. The \??\ prefix is a symbolic link inside the Object Manager namespace pointing to the current session's device namespace — equivalent to Win32's \\.\, but accessible only through the native API.
