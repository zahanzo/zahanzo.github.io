01

O Paradoxo dos 8 bytes

Todo programador C aprende que sizeof(char) == 1 e sizeof(int) == 4. É razoável, então, esperar que o struct abaixo ocupe 5 bytes:

Every C programmer learns that sizeof(char) == 1 and sizeof(int) == 4. It is reasonable, then, to expect the struct below to occupy 5 bytes:

the paradox c 
struct Paradox {
    char  a;  /* 1 byte  — you expect offset 0 */
    int   b;  /* 4 bytes — you expect offset 1 */
};

/* Expectation: sizeof(Paradox) == 1 + 4 == 5 */
/* Reality:     sizeof(Paradox) == 8           */

printf("sizeof(Paradox) = %zu\n", sizeof(struct Paradox)); // 8
printf("offsetof(a)     = %zu\n", offsetof(struct Paradox, a)); // 0
printf("offsetof(b)     = %zu\n", offsetof(struct Paradox, b)); // 4, not 1

  ▒ = padding byte (indeterminate value, NOT zero)
  b = bytes of the int member, starting at a 4-byte-aligned boundary

O compilador inseriu 3 bytes de padding entre a e b. Isso não é desperdício acidental — é o compilador obedecendo a uma restrição física do hardware: um int de 4 bytes deve começar em um endereço divisível por 4. O porquê disso está na forma como as CPUs leem memória.

The compiler inserted 3 bytes of padding between a and b. This is not accidental waste — it is the compiler obeying a physical hardware constraint: a 4-byte int must start at an address divisible by 4. The reason lies in how CPUs actually read memory.

02

A Realidade do Hardware

CPUs não leem memória byte a byte. O hardware opera em unidades maiores chamadas cache lines — no x86 moderno, sempre 64 bytes. Quando você acessa qualquer byte de memória, o processador carrega os 64 bytes inteiros da cache line que contém aquele byte. Isso significa que a posição de um dado dentro de uma cache line afeta diretamente a performance.

CPUs do not read memory byte by byte. The hardware operates in larger units called cache lines — on modern x86, always 64 bytes. When you access any byte of memory, the processor loads the entire 64-byte cache line containing that byte. This means a piece of data's position within a cache line directly affects performance.

// x86 vs ARM: falha silenciosa ou explosão ruidosa?

O comportamento de acessos não-alinhados difere radicalmente por arquitetura. No x86, o hardware tolera acessos não-alinhados — o microcode faz dois acessos ao barramento internamente, mas o programa não crasha. No ARM, muitas configurações geram um Data Abort imediato: o sistema operacional converte isso em SIGBUS e o processo morre.

The behavior of unaligned accesses differs radically by architecture. On x86, hardware tolerates unaligned accesses — microcode performs two bus accesses internally, but the program does not crash. On ARM, many configurations generate an immediate Data Abort: the OS converts this to SIGBUS and the process dies.

behavior by architecture — summary ascii 
  Architecture   Unaligned access behavior
  ─────────────  ──────────────────────────────────────────────────────────
  x86 / x86_64   Hardware handles it transparently.
                 Cost: 1 extra memory cycle if straddling a cache line.
                 Cost: ~10–20% throughput penalty in hot loops.
                 Atomic operations (LOCK prefix) REQUIRE alignment — violation
                 → #GP (General Protection Fault, i.e. SIGBUS/SIGSEGV).

  ARM Cortex-A   Default: fault (SIGBUS) on unaligned 16/32/64-bit access.
                 SCTLR.A bit (Alignment Check) controls this.
                 Many Linux kernels enable software fixup: trap → emulate →
                 slower than aligned, but program survives.

  ARM Cortex-M   ALWAYS faults on unaligned multi-byte access (HardFault).
                 No software fixup. Your firmware crashes, period.

  RISC-V         Spec allows implementations to fault on unaligned access.
                 Embedded implementations (RV32I without M ext): always fault.

  MIPS           Historically strict: always faults on unaligned access.

UB na linguagem, não só no hardware

O padrão C/C++ garante que variáveis e membros de struct são corretamente alinhados quando alocados normalmente. Se você força um acesso não-alinhado — por exemplo, fazendo cast de um char* para int* em um endereço ímpar — isso é Undefined Behavior pela linguagem, independente do hardware tolerar ou não. O compilador pode assumir que ponteiros são alinhados e gerar código que quebra silenciosamente em otimizações agressivas (-O2 / -O3).

The C/C++ standard guarantees that variables and struct members are properly aligned when allocated normally. If you force an unaligned access — for example, casting a char* to int* at an odd address — that is Undefined Behavior at the language level, regardless of whether the hardware tolerates it. The compiler may assume pointers are aligned and generate code that silently breaks under aggressive optimization (-O2 / -O3).

03

As Regras de Padding

O compilador insere padding seguindo regras definidas pela ABI da plataforma alvo. Para o System V AMD64 ABI (Linux/macOS x86_64) e para a maioria das ABIs modernas, as regras são:

The compiler inserts padding following rules defined by the target platform's ABI. For the System V AMD64 ABI (Linux/macOS x86_64) and most modern ABIs, the rules are:

natural alignment by type — System V AMD64 ABI ascii 
  Type                  Size    Alignment   Notes
  ────────────────────  ──────  ──────────  ─────────────────────────────────────
  char / uint8_t         1       1           no alignment requirement
  short / uint16_t       2       2           must be at even address
  int / uint32_t / float 4       4           must be at addr % 4 == 0
  long / uint64_t        8       8           must be at addr % 8 == 0
  double                 8       8           must be at addr % 8 == 0
  long double            16      16          (x87 80-bit padded to 16 on x86_64)
  pointer (64-bit)       8       8           must be at addr % 8 == 0
  struct                 varies  max(members) struct itself aligned to largest member

  Rule 1: each member is placed at the next address satisfying its alignment.
  Rule 2: padding is inserted BETWEEN members to satisfy the next member's alignment.
  Rule 3: trailing padding is added so sizeof(struct) is a multiple of the struct's alignment.

// Por que o trailing padding existe?

O trailing padding garante que um array de structs seja válido. Se você declara struct S arr[2], arr[1] precisa estar alinhado assim como arr[0]. Isso só é possível se sizeof(S) for múltiplo do alinhamento do struct. Sem o trailing padding, o segundo elemento do array estaria no endereço errado.

Trailing padding ensures that an array of structs is valid. If you declare struct S arr[2], arr[1] must be aligned just like arr[0]. This is only possible if sizeof(S) is a multiple of the struct's alignment. Without trailing padding, the second array element would be at the wrong address.

trailing padding to guarantee array alignment c 
struct Tricky {
    int   a;   /* 4 bytes @ offset 0 */
    char  b;   /* 1 byte  @ offset 4 */
               /* 3 bytes trailing padding @ offset 5, 6, 7 */
};
/* sizeof(Tricky) == 8 (not 5!) */
/* Alignment of Tricky == 4 (largest member = int) */
/* 8 is the smallest multiple of 4 that fits int(4) + char(1) */

struct Tricky arr[2];
/* arr[0] at addr 0x00: int @ 0x00 ✓ (4-aligned), char @ 0x04 ✓ */
/* arr[1] at addr 0x08: int @ 0x08 ✓ (4-aligned), char @ 0x0C ✓ */
/* Without trailing padding, arr[1] would be at 0x05 → int @ 0x05 ✗ */
04

Packing e Otimização

// Estratégia 1: ordenar membros do maior para o menor

A maneira mais simples e sem custo de eliminar padding é ordenar os membros do struct por alinhamento decrescente. Membros maiores primeiro eliminam a necessidade de padding interno, pois nenhum membro menor pode exigir alinhamento maior do que um membro que já o precede.

The simplest and zero-cost way to eliminate padding is to order struct members by decreasing alignment. Larger members first eliminate the need for internal padding, since no smaller member can require greater alignment than a member that already precedes it.

before and after reordering c 
/* BEFORE: random order — 32 bytes due to padding */
struct Unoptimized {
    char    a;       /*  1 byte  @ offset  0 */
                     /*  3 bytes padding    */
    int     b;       /*  4 bytes @ offset  4 */
    char    c;       /*  1 byte  @ offset  8 */
                     /*  7 bytes padding    */
    double  d;       /*  8 bytes @ offset 16 */
    short   e;       /*  2 bytes @ offset 24 */
                     /*  6 bytes padding    */
};                   /* sizeof == 32          */

/* AFTER: sorted largest-to-smallest — 16 bytes, zero internal padding */
struct Optimized {
    double  d;       /*  8 bytes @ offset  0 */
    int     b;       /*  4 bytes @ offset  8 */
    short   e;       /*  2 bytes @ offset 12 */
    char    a;       /*  1 byte  @ offset 14 */
    char    c;       /*  1 byte  @ offset 15 */
};                   /* sizeof == 16, 50% smaller */

// Estratégia 2: __attribute__((packed))

A diretiva __attribute__((packed)) (GCC/Clang) ou #pragma pack(1) (MSVC e outros) instrui o compilador a eliminar todo o padding, colocando os membros em bytes consecutivos. O resultado é o menor struct possível — mas com custo severo e riscos de UB. Use apenas quando o layout exato dos bytes é mandatório (protocolos de rede, formatos de arquivo binário, comunicação com hardware).

The __attribute__((packed)) directive (GCC/Clang) or #pragma pack(1) (MSVC and others) instructs the compiler to eliminate all padding, placing members at consecutive bytes. The result is the smallest possible struct — but with severe costs and UB risks. Use only when the exact byte layout is mandatory (network protocols, binary file formats, hardware register maps).

packed struct — size wins, safety loses c 
struct __attribute__((packed)) NetworkHeader {
    uint8_t   flags;    /* offset 0 — aligned ✓ */
    uint32_t  src_ip;   /* offset 1 — UNALIGNED on 4-byte boundary ✗ */
    uint32_t  dst_ip;   /* offset 5 — UNALIGNED ✗ */
    uint16_t  port;     /* offset 9 — UNALIGNED on 2-byte boundary ✗ */
};
/* sizeof == 11 (correct for wire format) */

/* SAFE: read via memcpy — compiler handles unaligned load correctly */
uint32_t ip;
memcpy(&ip, &hdr.src_ip, sizeof(ip));

/* DANGEROUS: direct pointer dereference — UB, crashes on ARM Cortex-M */
uint32_t ip_unsafe = hdr.src_ip;  /* compiler may load as 4-byte word → SIGBUS */

/* ALSO DANGEROUS: taking the address of a packed member */
uint32_t *ptr = &hdr.src_ip;  /* GCC warns: address of packed member */
*ptr = htonl(0xC0A80001);     /* UB: unaligned write via aligned pointer */

// Estratégia 3: alignas()

alignas(N) (C11/C++11) força um alinhamento maior que o natural — útil para garantir alinhamento de cache line (alignas(64)) e evitar false sharing, ou para garantir alinhamento de 16/32 bytes exigido por instruções SIMD (SSE/AVX).

alignas(N) (C11/C++11) forces greater alignment than natural — useful for guaranteeing cache line alignment (alignas(64)) and avoiding false sharing, or for ensuring the 16/32-byte alignment required by SIMD instructions (SSE/AVX).

alignas — SIMD and false sharing prevention cpp 
#include <stdalign.h>   /* C11 */
#include <immintrin.h>  /* SSE/AVX intrinsics */

/* Force 16-byte alignment for SSE2 _mm_load_ps (requires 16-byte alignment) */
struct alignas(16) Vec4 {
    float x, y, z, w;   /* 16 bytes, 16-byte aligned → _mm_load_ps safe */
};

/* Force 32-byte alignment for AVX _mm256_load_ps */
struct alignas(32) Vec8 {
    float v[8];          /* 32 bytes, 32-byte aligned → _mm256_load_ps safe */
};

/* Prevent false sharing in multi-threaded code:                          */
/* Two threads writing adjacent fields → both dirty the SAME cache line → */
/* hardware must synchronize the line on every write (cache coherency IPI) */
struct ThreadCounter {
    alignas(64) uint64_t count;  /* occupies its own 64-byte cache line */
    alignas(64) uint64_t errors; /* different cache line → no false sharing */
};
/* sizeof(ThreadCounter) == 128 (two cache lines) */
/* Without alignas(64): sizeof == 16, both fields share ONE cache line  */

Exemplos Completos

O exemplo abaixo reúne os cenários mais comuns. Compilado com GCC 13 no Linux x86_64 (-O0, System V AMD64 ABI).

The example below covers the most common scenarios. Compiled with GCC 13 on Linux x86_64 (-O0, System V AMD64 ABI).

alignment_demo.c — all cases in one file c 
#include <stdio.h>
#include <stddef.h>    /* offsetof */
#include <stdint.h>

/* ─────────────────────────────────────────────────────────────────── */
/* CASE 1: naive order — heavy padding                                 */
struct Naive {
    char   a;     /* 1 byte  */
    double b;     /* 8 bytes */
    int    c;     /* 4 bytes */
    char   d;     /* 1 byte  */
    short  e;     /* 2 bytes */
};
/* sizeof == 32 (14 bytes of data, 18 bytes of padding!) */

/* ─────────────────────────────────────────────────────────────────── */
/* CASE 2: optimal order — no internal padding                         */
struct Optimal {
    double b;     /* 8 bytes */
    int    c;     /* 4 bytes */
    short  e;     /* 2 bytes */
    char   a;     /* 1 byte  */
    char   d;     /* 1 byte  */
};
/* sizeof == 16 (14 bytes of data, 2 bytes trailing padding) */

/* ─────────────────────────────────────────────────────────────────── */
/* CASE 3: packed — minimum size, maximum danger                       */
struct __attribute__((packed)) Packed {
    char   a;     /* 1 byte  */
    double b;     /* 8 bytes, UNALIGNED at offset 1 */
    int    c;     /* 4 bytes, UNALIGNED at offset 9 */
    char   d;     /* 1 byte  */
    short  e;     /* 2 bytes, UNALIGNED at offset 14 */
};
/* sizeof == 16 (14 bytes of data, 0 padding, all unaligned except a/d) */

int main(void) {
    printf("=== struct Naive ===\n");
    printf("  sizeof   = %zu\n", sizeof(struct Naive));
    printf("  offsetof(a) = %zu\n", offsetof(struct Naive, a));
    printf("  offsetof(b) = %zu\n", offsetof(struct Naive, b));
    printf("  offsetof(c) = %zu\n", offsetof(struct Naive, c));
    printf("  offsetof(d) = %zu\n", offsetof(struct Naive, d));
    printf("  offsetof(e) = %zu\n", offsetof(struct Naive, e));

    printf("\n=== struct Optimal ===\n");
    printf("  sizeof   = %zu\n", sizeof(struct Optimal));
    return 0;
}

// A regra prática definitiva

Para qualquer struct que será alocada em grande quantidade (arrays, pools, caches), aplique sempre: (1) ordene membros do maior para o menor alinhamento, (2) agrupe membros do mesmo tipo quando possível, (3) verifique com sizeof() e pahole. Use __packed__ ou #pragma pack apenas quando o layout binário exato é um requisito do protocolo — e sempre acesse membros não-alinhados via memcpy(), nunca por dereference direto.

For any struct that will be allocated in large quantities (arrays, pools, caches), always apply: (1) order members from largest to smallest alignment, (2) group members of the same type when possible, (3) verify with sizeof() and pahole. Use __packed__ or #pragma pack only when the exact binary layout is a protocol requirement — and always access unaligned members via memcpy(), never by direct dereference.
