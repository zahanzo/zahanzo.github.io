01

Arquitetura e Layout de Memória

O espaço de endereçamento de um processo em x86_64 é dividido em regiões com propósitos distintos. A stack ocupa a região de endereços altos do espaço virtual e cresce em direção aos endereços mais baixos a cada alocação. Esse comportamento é contra-intuitivo à primeira vista, mas é uma convenção de hardware que remonta às primeiras arquiteturas x86.

A process's address space in x86_64 is divided into regions with distinct purposes. The stack occupies the high-address region of virtual space and grows toward lower addresses with each allocation. This behavior is counterintuitive at first, but it is a hardware convention that dates back to the earliest x86 architectures.

layout de memória de um processo (simplificado) ascii 
  0xFFFFFFFFFFFFFFFF  ┌─────────────────────────┐
                      │   kernel space          │  (inaccessible to userland)
  0xFFFF800000000000  ├─────────────────────────┤
                      │                         │
                      │   stack  ↓ grows down   │  ← RSP starts here (high addr)
                      │   ...                   │
                      ├─────────────────────────┤
                      │   (unmapped gap)        │
                      ├─────────────────────────┤
                      │   heap   ↑ grows up     │  ← malloc/new live here
                      ├─────────────────────────┤
                      │   BSS  (uninit globals) │
                      ├─────────────────────────┤
                      │   data (init globals)   │
                      ├─────────────────────────┤
                      │   text (code / .text)   │
  0x0000000000400000  └─────────────────────────┘

// RSP — Stack Pointer

RSP é o registrador de 64 bits que aponta sempre para o topo atual da stack — o endereço mais baixo atualmente ocupado. Cada push decrementa RSP em 8 bytes e grava o valor; cada pop lê o valor e incrementa RSP em 8 bytes. O hardware do processador atualiza RSP automaticamente nessas instruções.

RSP is the 64-bit register that always points to the current top of the stack — the lowest currently occupied address. Each push decrements RSP by 8 bytes and writes the value; each pop reads the value and increments RSP by 8 bytes. The processor hardware updates RSP automatically on those instructions.

push / pop mechanics asm 
; PUSH: decrements RSP first, then writes
push  rax
; equivalent to:
sub   rsp, 8
mov   [rsp], rax

; POP: reads first, then increments RSP
pop   rax
; equivalent to:
mov   rax, [rsp]
add   rsp, 8

// RBP — Base Pointer (Frame Pointer)

RBP aponta para a base do stack frame atual — o ponto de referência fixo de uma função. Enquanto RSP flutua com cada push/pop, RBP permanece estático durante toda a execução de uma função. Isso permite que o compilador acesse variáveis locais e argumentos por offsets fixos em relação a RBP (e.g., [rbp-8], [rbp-16]), independentemente do estado de RSP.

RBP points to the base of the current stack frame — the fixed reference point for a function. While RSP fluctuates with every push/pop, RBP stays static for the entire duration of a function's execution. This allows the compiler to access local variables and arguments via fixed offsets relative to RBP (e.g., [rbp-8], [rbp-16]), regardless of the current state of RSP.

02

Stack Frame: Prólogo e Epílogo

Um stack frame é a região da stack reservada para uma única invocação de função. Ele contém o endereço de retorno, o RBP do chamador salvo, as variáveis locais e, em alguns casos, argumentos extras. A criação e destruição de um frame seguem um protocolo rígido denominado prólogo e epílogo.

A stack frame is the stack region reserved for a single function invocation. It contains the return address, the saved caller's RBP, local variables, and in some cases extra arguments. Frame creation and destruction follow a strict protocol called the prologue and epilogue.

// PRÓLOGO (entrada da função)
push rbp — salva o RBP do chamador
mov rbp, rsp — RBP aponta para o topo atual
sub rsp, N — reserva espaço para variáveis locais
// EPÍLOGO (saída da função)
leave — atalho para as duas instruções abaixo
mov rsp, rbp — descarta variáveis locais
pop rbp — restaura o RBP do chamador
ret — salta para o endereço de retorno
layout do stack frame durante execução de foo() ascii 
  ; High addresses (caller's territory)
  ┌──────────────────────────────────┐
  │  caller's local variables / args │
  ├──────────────────────────────────┤
  │  return address (pushed by CALL) │  ← [rbp + 8]
  ├──────────────────────────────────┤ ←  RBP  (frame base)
  │  saved RBP  (caller's frame ptr) │  ← [rbp + 0]
  ├──────────────────────────────────┤
  │  local var  a   (8 bytes)        │  ← [rbp - 8]
  ├──────────────────────────────────┤
  │  local var  b   (8 bytes)        │  ← [rbp - 16]
  ├──────────────────────────────────┤
  │  local var  c   (8 bytes)        │  ← [rbp - 24]
  ├──────────────────────────────────┤ ←  RSP  (stack top, grows ↓)
  │  (unmapped)                      │
  └──────────────────────────────────┘
  ; Low addresses

Por que salvar RBP?

Ao entrar em foo(), o registrador RBP ainda contém o valor do frame anterior (o chamador). Se simplesmente sobrescrevermos RBP com RSP, perdemos o contexto do chamador. O push rbp preserva esse valor na stack antes de RBP ser redirecionado. Na saída, o pop rbp restaura exatamente esse valor, reconstruindo a cadeia de frames — que é o que debuggers usam para gerar stack traces.

When entering foo(), the RBP register still holds the previous frame's value (the caller's). If we simply overwrite RBP with RSP, we lose the caller's context. The push rbp preserves that value on the stack before RBP is redirected. On exit, pop rbp restores exactly that value, reconstructing the frame chain — which is what debuggers use to generate stack traces.

03

Gerenciamento de Dados na Stack

// Variáveis Locais

O compilador determina em tempo de compilação o tamanho total das variáveis locais de uma função e emite um único sub rsp, N no prólogo para reservar todo esse espaço de uma vez. Cada variável recebe um offset fixo em relação a RBP. Não há instrução por variável; o compilador simplesmente usa o endereço computado.

The compiler determines at compile time the total size of a function's local variables and emits a single sub rsp, N in the prologue to reserve all that space at once. Each variable is assigned a fixed offset relative to RBP. There is no per-variable instruction; the compiler simply uses the computed address.

acesso a variáveis locais via RBP asm 
sub   rsp, 24          ; reserve 3 × 8 bytes for a, b, c

mov   QWORD PTR [rbp-8],  10  ; a = 10
mov   QWORD PTR [rbp-16], 20  ; b = 20
mov   rax,  [rbp-8]
add   rax,  [rbp-16]
mov   QWORD PTR [rbp-24], rax  ; c = a + b

// Endereço de Retorno

A instrução call foo realiza duas operações atomicamente: empurra o endereço da próxima instrução (o ponto de retorno) na stack, e então salta para foo. Quando ret é executado, ele faz o caminho inverso: lê o endereço do topo da stack e salta para ele. Esse endereço fica em [rbp + 8] dentro do frame da função chamada.

The call foo instruction performs two operations atomically: it pushes the address of the next instruction (the return point) onto the stack, then jumps to foo. When ret executes, it does the reverse: reads the address from the top of the stack and jumps to it. That address lives at [rbp + 8] inside the called function's frame.

call / ret mechanics asm 
; CALL: push return address, then jump
call  foo
; equivalent to:
push  rip            ; (conceptual — RIP is the next instruction's addr)
jmp   foo

; RET: pop return address and jump to it
ret
; equivalent to:
pop   rip            ; (conceptual — loads address back into RIP)

C vs. Assembly: a tradução do compilador

O exemplo abaixo mostra a tradução de uma função C simples para Assembly x86_64, compilado com gcc -O0 (sem otimizações, para preservar o stack frame explícito) no Linux com a ABI System V.

The example below shows the translation of a simple C function to x86_64 Assembly, compiled with gcc -O0 (no optimizations, to preserve the explicit stack frame) on Linux with the System V ABI.

// C source
#include <stdio.h>

int add(int a, int b) {
    int result = a + b;
    return result;
}

int main(void) {
    int x = add(3, 7);
    return 0;
}
// gcc -O0 x86_64 output (System V)
add:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 16
    mov    [rbp-20], edi  ; spill param a
    mov    [rbp-24], esi  ; spill param b
    mov    edx, [rbp-20]
    mov    eax, [rbp-24]
    add    eax, edx
    mov    [rbp-4],  eax  ; result = a + b
    mov    eax, [rbp-4]   ; return value → EAX
    leave
    ret

main:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 16
    mov    esi, 7          ; arg b → RSI (2nd arg)
    mov    edi, 3          ; arg a → RDI (1st arg)
    call   add
    mov    [rbp-4],  eax  ; x = return value
    mov    eax, 0
    leave
    ret
estado da stack durante add(3, 7) ascii 
  ; --- Inside main(), just before CALL add ---
  ┌──────────────────────────────────────┐ ← main's RBP
  │  saved RBP  (program's frame ptr)    │
  ├──────────────────────────────────────┤
  │  x  (int, 4 bytes, uninitialized)    │  [rbp-4]
  │  (padding / alignment)              │
  ├──────────────────────────────────────┤ ← main's RSP
  │  RDI=3, RSI=7 already loaded        │  (registers, not stack)

  ; --- After CALL add (inside add's prologue) ---
  ┌──────────────────────────────────────┐ ← main's frame (above)
  │  return address  (next instr in main)│  ← [rbp+8]  (pushed by CALL)
  ├──────────────────────────────────────┤ ← add's RBP (after push rbp + mov rbp,rsp)
  │  saved RBP  (main's frame ptr)       │  [rbp+0]
  ├──────────────────────────────────────┤
  │  result  (int)                       │  [rbp-4]
  │  (alignment padding)                 │
  │  a  (spilled from EDI)               │  [rbp-20]
  │  b  (spilled from ESI)               │  [rbp-24]
  ├──────────────────────────────────────┤ ← add's RSP

Convenções de Chamada — System V AMD64 ABI

A ABI (Application Binary Interface) define o contrato entre quem chama e quem é chamado. Em Linux e macOS x86_64, a System V AMD64 ABI especifica onde os argumentos devem estar, quem preserva quais registradores, e como o valor de retorno é comunicado.

The ABI (Application Binary Interface) defines the contract between caller and callee. On Linux and macOS x86_64, the System V AMD64 ABI specifies where arguments must be placed, which registers each side must preserve, and how the return value is communicated.

// Passagem de Argumentos Inteiros/Ponteiros

ordem dos registradores para os primeiros 6 argumentos asm 
; int func(a,     b,     c,     d,     e,     f,     g,   h)
;          RDI    RSI    RDX    RCX    R8     R9    [stack+8] [stack+16]

; The 7th argument onwards are pushed on the stack, right-to-left.
; The return value is in RAX (or RDX:RAX for 128-bit values).

example_call:
    mov  rdi, 1      ; arg 1
    mov  rsi, 2      ; arg 2
    mov  rdx, 3      ; arg 3
    mov  rcx, 4      ; arg 4
    mov  r8,  5      ; arg 5
    mov  r9,  6      ; arg 6
    push 8          ; arg 8 pushed first (right-to-left)
    push 7          ; arg 7
    call func
    add  rsp, 16    ; caller cleans up stack args

// Registradores: quem preserva o quê?

callee-saved vs caller-saved (System V AMD64) ascii 
  ; CALLER-SAVED (volatile) — callee may destroy these freely:
  RAX  RCX  RDX  RSI  RDI  R8  R9  R10  R11
  ; The CALLER must save these before CALL if it needs them after.

  ; CALLEE-SAVED (non-volatile) — callee MUST restore before RET:
  RBX  RBP  R12  R13  R14  R15  RSP
  ; The CALLEE must push/pop these if it uses them.

Stack Alignment

A System V ABI exige que RSP esteja alinhado a 16 bytes no momento do call. Como o próprio call empurra 8 bytes (o endereço de retorno), RSP dentro da função estará desalinhado em 8. Por isso o prólogo usa sub rsp, N onde N é múltiplo de 16 — para restaurar o alinhamento antes de qualquer instrução SSE/AVX que exija memória alinhada.

The System V ABI requires RSP to be 16-byte aligned at the point of call. Since call itself pushes 8 bytes (the return address), RSP inside the function will be misaligned by 8. That is why the prologue uses sub rsp, N where N is a multiple of 16 — to restore alignment before any SSE/AVX instruction that requires aligned memory.

Segurança: Stack Buffer Overflow

Compreender a stack não é apenas um exercício teórico — é o fundamento de uma das classes de vulnerabilidades mais exploradas da história da computação. O Stack Buffer Overflow ocorre quando dados são escritos em um buffer local além dos seus limites, sobrescrevendo regiões adjacentes do frame.

Understanding the stack is not merely a theoretical exercise — it is the foundation of one of the most exploited vulnerability classes in computing history. A Stack Buffer Overflow occurs when data is written into a local buffer beyond its bounds, overwriting adjacent regions of the stack frame.

// O código vulnerável

função vulnerável — sem checagem de bounds c 
#include <string.h>

void vulnerable(char *input) {
    char buffer[64];          // 64 bytes allocated on stack
    strcpy(buffer, input);    // NO bounds check — classic mistake
}                             // if len(input) > 64: overflow!

// O que acontece na memória

stack antes e depois do overflow ascii 
  ; --- BEFORE overflow (normal state) ---
  ┌──────────────────────────────────────┐
  │  return address   0x00400512         │  ← [rbp+8]  legitimate
  ├──────────────────────────────────────┤ ← RBP
  │  saved RBP        0x7fffffffe3b0     │
  ├──────────────────────────────────────┤
  │  buffer[0..63]    "AAAA..."          │  64 bytes
  ├──────────────────────────────────────┤ ← RSP

  ; --- AFTER overflow (input = 80 bytes of 'A' + attacker addr) ---
  ┌──────────────────────────────────────┐
  │  return address   0x4141414141414141 │  ← OVERWRITTEN with 'AAAA...'
  ├──────────────────────────────────────┤ ← RBP (also corrupted)
  │  saved RBP        0x4141414141414141 │  ← OVERWRITTEN
  ├──────────────────────────────────────┤
  │  buffer[0..63]    "AAAA...AAAA..."   │  ← filled
  │  buffer[64..71]   "AAAAAAAA"         │  ← overflows into saved RBP
  │  buffer[72..79]   0xDEADBEEFCAFEBABE │  ← overwrites return address!
  ├──────────────────────────────────────┤

Quando ret é executado, o processador carrega 0xDEADBEEFCAFEBABE em RIP e continua a execução nesse endereço — que pode ser código arbitrário do atacante (shellcode), outro ponto do próprio binário (ret2libc), ou uma cadeia de gadgets ROP (Return-Oriented Programming).

When ret executes, the processor loads 0xDEADBEEFCAFEBABE into RIP and continues execution there — which may be the attacker's arbitrary code (shellcode), another point in the binary (ret2libc), or a chain of ROP gadgets (Return-Oriented Programming).

// Mitigações Modernas

camadas de defesa no kernel/compilador/hardware ascii 
  Stack Canary    │ gcc -fstack-protector
                  │ A random value placed between locals and return address.
                  │ Checked before RET — mismatch aborts the process.
                  │ Bypassed by: info leak + overwrite with correct canary.

  NX / DEP        │ Hardware (Intel XD bit / AMD NX bit)
                  │ Marks stack pages as non-executable.
                  │ Shellcode injected on stack cannot be executed.
                  │ Bypassed by: ROP (reuses existing executable code).

  ASLR            │ Kernel feature (randomize_va_space)
                  │ Randomizes base addresses of stack, heap, and libraries.
                  │ Attacker cannot predict where to redirect RIP.
                  │ Bypassed by: info leak leaking a pointer into the space.

  CFI / Shadow Stack │ Intel CET (Control-flow Enforcement Technology)
                     │ Hardware maintains a parallel shadow stack for return addrs.
                     │ RET is validated against the shadow stack — overwrite fails.

Por que ainda importa em 2026?

Stack overflows em código C/C++ sem mitigações ainda são comuns em firmware, sistemas embarcados, drivers e código legado. Mais importante: entender o mecanismo é pré-requisito para compreender técnicas avançadas como ROP chains, heap feng shui e bypass de ASLR — que são a base do exploit development moderno e, pelo lado defensivo, da análise de vulnerabilidades.

Stack overflows in unmitigated C/C++ code remain common in firmware, embedded systems, drivers, and legacy code. More importantly, understanding the mechanism is a prerequisite for advanced techniques such as ROP chains, heap feng shui, and ASLR bypass — which form the foundation of modern exploit development and, on the defensive side, of vulnerability research.
